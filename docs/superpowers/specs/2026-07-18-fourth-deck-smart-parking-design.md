# 第四个项目经历 Deck：智慧停车 2.0 设计 Spec

> 日期：2026-07-18
>
> 状态：设计已确认，等待用户复核书面 Spec
>
> 页面定位：个人网站「项目经历」第四个公开案例；以外包停车系统内收和持续演进能力重建为主线

## 1. 背景

停车是综合商场顾客到场、消费和离场体验中的重要基础服务，也连接车辆、访客、会员、积分、卡券、订单和支付。停车系统能否稳定维护和持续迭代，会同时影响顾客体验、商场运营和内部业务协同。

停车 1.0 由外包团队开发，原系统仍能支撑既有业务，但外包响应慢、需求迭代慢，后台与小程序的 UI 和交互存在明显不合理之处；会员、积分、卡券和订单模型也与公司内部体系割裂，需要额外建设专用接口。真正需要解决的不是“重做一套停车页面”，而是公司逐渐失去对核心停车产品的定义权、维护能力和演进控制。

迁移时没有可直接使用的产品文档和开发技术文档。本人以仍在运行的 1.0 后台和小程序为主要事实来源，逐页面、逐流程反推出既有功能、规则、状态和异常分支，再接近完整功能覆盖地重新规划 2.0。2.0 保留已经被业务验证的能力，但不照搬旧页面、历史硬规则或旧系统分层。

本 Deck 应证明：系统内收的价值不只发生在首次上线时，更体现在 XCU—MCU 身份解耦、多层停车接入、权益与支付生命周期重构，以及上线后对双重支付、车辆滥用、退款和跨日核销等真实问题的持续响应。

## 2. 已确认的核心决策

### 2.1 第一记忆点

访客看完后应能复述：

> 不是重做一套停车页面，而是在缺少产品与技术文档的情况下逆向还原外包系统，保留业务精华、重做产品与系统关系，让停车重新拥有持续演进能力。

多层架构、身份解耦、权益编排、交易状态和异常治理都是这项判断的证据，不与主线争抢。

### 2.2 叙事路线

采用「业务重要性 → 失去演进能力 → 黑盒还原 → 原生重构 → 持续迭代 → 真实迁移边界」路线：

1. 先说明停车为什么是综合商场的重要服务，而不是边缘功能。
2. 说明 1.0 仍在运行，但外包响应、迭代、体验和内部业务耦合方式已经限制继续演进。
3. 说明在无可用文档的情况下，如何通过线上后台和小程序逆向还原完整能力边界。
4. 以 XCU—MCU 身份解耦证明 2.0 不是原样复制：访客无需强制开卡即可缴费，会员权益成为自愿进入的支线。
5. 以停车后端、停车中台和外部适配分层，以及权益编排和冻结生命周期证明系统深度。
6. 以上线后的真实问题和迭代轨迹证明内部团队重新获得持续响应能力。
7. 用 8 个 2.0 商场的真实运行和 27 个存量商场仍未迁移共同呈现落地结果与迁移现实。

### 2.3 不是一比一复制

- 1.0 后台和小程序是功能盘点的主要事实来源，不是需要照抄的页面模板。
- 2.0 的功能覆盖接近 1.0，但产品结构、业务规则、交互和系统分层重新规划。
- 保留经过业务验证的能力，舍弃不合理体验和低收益历史硬规则。
- 消费减免不继续复制 1.0 对消费时间和停车场次的大量复杂匹配；当前按每名会员每天最多使用一次处理低频二次入场场景。
- UI 与交互优化是迁移中需要解决的问题之一，但不是案例的核心成果。

### 2.4 首发产品决策

XCU—MCU 身份解耦属于 2.0 首次上线能力，而不是上线后的补丁：

```text
停车 1.0
查询车费 → 强制开卡 / 手机号授权 → 会员身份 → 缴费

停车 2.0
查询车费 → XCU 访客身份 → 直接缴费
                         ↓ 用户自愿成为会员
                       MCU 会员身份 → 使用会员权益
```

- XCU 承接未开卡的小程序访客、车辆查询和停车支付。
- 用户无需开卡即可支付停车费，避免把会员卡或手机号授权设为支付前置条件。
- 用户自愿成为会员后，再通过 MCU 使用会员等级、积分和卡券权益。
- 该设计既满足支付宝平台的授权规则，也把停车恢复为面向所有顾客的基础服务。
- 对外不展示“未开卡支付占比”或转化提升，因为当前没有相应结果数据，也不需要用占比证明规则合规。

### 2.5 结果证明与迁移现实

结尾使用「真实运行规模 + 渐进迁移边界」：

- 停车系统共覆盖 35 个商场，其中 8 个运行 2.0，27 个继续运行 1.0。
- 8 个 2.0 商场均启用完整缴费和权益能力。
- 按现有月均口径，8 个 2.0 商场约产生 7.57 万笔月停车缴费单。
- 某集团 19 个商场共用集团停车中台，其中 1 个运行 2.0，18 个继续运行 1.0。
- 27 个存量商场涉及车辆、订单、规则和历史数据整合成本，不能包装成 2.0 已经全面替代 1.0。
- 退款能力已经上线，但尚未处理真实线上退款订单，不能包装成大量采用或效率成果。

S2 的业务背景数据与 S10 的 2.0 落地数据必须分开。S2 证明“停车为什么重要”，S10 证明“2.0 实际走到了哪里”，不在两屏重复使用同一组金额和订单数据。

## 3. 目标与非目标

### 3.1 目标

- 让陌生访客先理解停车对综合商场的重要性，再理解为什么外包系统内收值得作为独立项目。
- 证明本人能够在缺少产品和技术文档时，从线上系统逆向还原业务能力并重新完成产品规划。
- 通过 XCU—MCU 身份解耦、内部业务接入、权益编排和支付生命周期证明产品与系统边界设计能力。
- 以上线后的持续迭代证明内收后恢复了问题定位、需求响应和异常治理能力。
- 用真实运行规模和版本并存说明项目已经落地，同时坦诚迁移成本和验证边界。
- 与销售线索的连续演进叙事、企业权限的模型对照叙事保持明显体验差异。

### 3.2 非目标

- 不写成停车厂商接口、数据库、支付中台或退款实现教程。
- 不把 1.0 描述成完全不可用或业务失败。
- 不把功能覆盖接近原系统写成旧页面、旧规则或旧代码的一比一复制。
- 不把 UI 改版作为系统内收的主要价值。
- 不声称 XCU—MCU 解耦提升了开卡率、支付转化或某类用户占比。
- 不把 35 个停车商场的规模全部归入 2.0。
- 不把退款能力写成已有真实订单采用结果。
- 不把 27 个商场尚未迁移包装成全面替代或强制迁移。
- 不使用生产后台截图、真实车牌、订单、客户、接口地址或未经核验的经营数据。
- 不使用 GSAP、长时间吸顶、横向滚动或汽车广告式运镜。
- 不修改其他项目 Deck 的内容、事实或动效。

## 4. 受众与成功标准

### 4.1 主要受众

- 招聘方和业务负责人：判断候选人能否接手历史包袱较重、跨系统且需要持续运营的复杂产品。
- 产品负责人：判断问题盘点、业务取舍、身份与权益建模、异常闭环和迁移策略能力。
- 研发与架构协作者：判断产品方案是否尊重内部业务模型、外部系统差异和交易状态边界。

### 4.2 内容成功标准

- 访客能说出停车 2.0 为什么不是一次 UI 改版。
- 访客能理解“无文档黑盒还原”与“原样复制旧系统”的区别。
- 访客能理解 XCU 访客无需开卡即可缴费，MCU 负责会员权益关系。
- 访客能理解停车后端、停车中台和外部适配的职责差异。
- 访客能理解权益叠加、会员与车辆双维度限制、冻结、支付和回退形成完整生命周期。
- 访客能从上线后迭代理解“恢复演进能力”的实际含义。
- 访客不会把 35 个商场全部误解为 2.0，也不会把退款能力误解为已有真实采用数据。

### 4.3 体验成功标准

- 11 个场景按纵向顺序完整可读。
- “车道导视与闸机”作为视觉隐喻服务叙事，不喧宾夺主。
- 桌面端、移动端、减少动态和脚本失败状态均不丢失语义。
- 深浅色主题下，琥珀车道线、青绿色系统节点、标签、边框和文字共同表达状态，颜色不是唯一编码。

## 6. 事实、状态与公开口径

### 6.1 本人角色

- 1.0 外包系统属于既有基础，不归入本人建设成果。
- 本人以 1.0 线上后台和小程序为主要事实来源，完成既有功能、流程、规则和状态的产品盘点。
- 2.0 的迁移范围、产品结构、系统分层、停车中台与外部适配方式、缴费主链、身份解耦、权益资格和冻结机制均由本人主导规划设计并推动上线。
- 会员等级、积分、停车券、消费减免、便捷出场、车辆风控、后台和小程序，以及后续退款与核销异常迭代均由本人持续负责。
- 实际工程开发由研发团队完成，不写成个人独立完成代码实现。

### 6.2 已上线事实

- 智慧停车 2.0 主系统已经上线。
- XCU—MCU 身份解耦在 2.0 首次上线时已经具备。
- 会员等级、积分抵现、停车券、消费减免和便捷出场已经上线。
- 车辆管理、防二次支付、停车退款和跨日核销等后续迭代已经上线。
- 8 个 2.0 商场均启用完整缴费和权益能力。

### 6.3 结果边界

- 2.0 的采用规模只能使用 8 个商场的数据，不能混用 27 个 1.0 商场。
- 约 7.57 万月停车缴费单属于 8 个 2.0 商场的现有月均汇总口径。
- 一个 2.0 商场刚上线，当前交易数据较少；实现时应与稳定运营项目分开说明，不能把不完整月份包装为成熟月均。
- 双重支付优化后未再明显发生同类问题，可以作为定性结果；没有统一前后异常单量时，不展示虚构下降比例。
- 退款和优惠回退能力已经上线，但尚未处理真实线上退款订单。
- 系统迁回内部后，问题定位、修复和需求响应更敏捷，但没有统一周期数据时不展示虚构效率比例。

### 6.4 脱敏规则

- 客户、集团、商场、停车厂商、外部系统、用户、车牌、订单和交易明细全部匿名。
- 后台、小程序和流程图使用 CSS/SVG 与虚构数据重新绘制。
- 不公开真实接口地址、外部系统地址、生产字段、完整优惠阈值或内部适配代码。
- 支付平台规则只说明授权边界，不直接使用带真实业务信息的违规截图。

## 7. S2 业务背景数据闸门

S2 需要用业务规模证明停车为什么重要，但当前可视化伴侣中的数字仅为排版示例，不是真实项目数据，禁止进入正式页面。

### 7.1 优先查询数据

周一由开发按统一口径查询：

1. 全停车业务或代表性商场的月均进场车次。
2. 同一统计范围的月均独立车牌数。
3. 同一统计范围的月停车收入或应付金额，并明确是现金实收还是优惠前应付。
4. 统计时间优先使用最近 3 个完整自然月。
5. 如果使用代表性商场数据，必须标明商场范围和选择原因；如果使用全停车业务，必须明确是否同时包含 1.0 与 2.0。

可选补充数据包括支付成功率、便捷出场零元缴费单量、XCU 未开卡访客缴费单量和双重支付优化前后的异常单量。可选数据只有在口径稳定且能改变页面判断时才进入 Deck。

### 7.2 生产页面行为

正式实现时必须满足以下一项：

- 使用经过开发核验、用户确认且标明范围与时间的真实数据。
- 如果实现时仍未获得可靠数据，S2 只展示顾客旅程与业务关系图，不展示任何占位数字。

S2 不直接复用 8 个 2.0 商场的月缴费单和应付金额作为背景数据，避免与 S10 的落地证据重复。

## 8. 逐屏内容设计

### S1 封面：让一套停车系统，重新拥有演进能力

- 标题直接建立项目价值，不使用“停车页面改版”或“系统重写”作为主标题。
- 副标题说明从无文档外包系统逆向还原，到重新规划产品、接入内部业务体系并持续迭代。
- 视觉上出现一条从 1.0 黑盒驶向 2.0 内部体系的车道，但不在封面展开完整架构。

### S2 业务背景：停车是商场体验的第一站与最后一站

- 说明停车贯穿顾客到场、消费和离场。
- 说明停车连接车辆、访客、会员、权益、订单和支付。
- 使用「规模先行」版式：三项经过核验的真实业务规模位于上方，顾客旅程位于下方。
- 如果真实数据未通过闸门，则取消数字卡片，扩大顾客旅程与业务关系图。
- 本屏结论是：当核心服务依赖响应慢、难迭代的外包系统，影响的不只是一个页面。

### S3 1.0 困境：系统仍在运行，但已经难以继续演进

- 1.0 能够支撑原有停车业务，不能描述为完全不可用。
- 外包响应慢、需求迭代慢，问题定位和修复依赖外部团队。
- 后台和小程序的 UI、交互存在不合理之处。
- 内部会员、积分、卡券和订单模型与停车系统割裂，需要额外专用接口。
- 本屏结论是公司逐渐失去对停车产品的演进控制。

### S4 黑盒还原：没有文档，只能从线上系统反推产品

- 没有可直接使用的产品文档和开发技术文档。
- 以仍在运行的 1.0 后台和小程序为主要事实来源。
- 逐项还原页面、流程、规则、状态、异常和能力边界。
- 功能覆盖接近原系统，但产品结构、业务规则、交互与系统分层重新规划。
- 视觉上从 1.0 黑盒中逐项照亮能力清单，表达“盘点”而不是“读取旧代码”。

### S5 首发转折：支付不再以开卡为前提

- 对照 1.0 的强制开卡 / 手机号授权链路和 2.0 的 XCU 访客支付链路。
- XCU 使用“未开卡的小程序访客身份”解释，MCU 使用“统一会员身份”解释。
- 用户无需开卡即可完成停车缴费。
- 用户自愿成为会员后，才通过 MCU 使用会员等级、积分和卡券权益。
- 视觉上闸机为 XCU 直行车道打开，MCU 是可选择进入的权益支线。

### S6 原生体系：把停车接回内部业务与统一外部协议

```text
XCU / MCU / 会员等级 / 积分 / 卡券 / SO / 支付
                         ↓
                    停车后端服务
                         ↓
                    停车业务中台
                         ↓
               厂商适配器 / 集团停车中台
                         ↓
                     商场停车场
```

- 停车后端负责内部业务模块与缴费、权益和订单状态编排。
- 停车中台定义统一查询、缴费通知、车辆和出场回调协议。
- 外部厂商差异由适配层承接，上层产品不重复理解每套协议。
- 不声称外部生态已经完全标准化；独立部署系统仍可能逐客户适配。
- 某集团自建停车中台可通过一次集团级对接覆盖 19 个商场，作为平台化适配示例。

### S7 权益编排：四类权益进入同一套可解释规则

- 一次缴费可以叠加会员等级、积分抵现、停车券和消费减免。
- 商场配置单次总减免上限。
- 手动缴费时，用户选择顺序决定优惠优先级。
- 前端负责即时展示优惠金额和剩余可优惠金额，后端重新校验资格、限制和总额度。
- 会员与车辆分别配置日、周、月上限；任一维度达到上限，本次权益不可继续使用。
- 车辆维度来自多人绑定同一车辆、轮流使用会员权益的真实风控问题。

### S8 交易生命周期：校验、冻结、支付、扣减与回退

```text
提交缴费
  → 后端重新校验权益与优惠额度
  → 冻结积分、停车券等权益
  → 创建待支付缴费单
     → 实付金额大于 0：支付成功后正式扣减 / 核销
     → 实付金额等于 0：直接扣减权益并完成缴费
     → 失败、取消或超时：解冻权益
  → 通知停车中台与客户停车系统
```

- 说明后端为何不能信任前端优惠结果。
- 说明权益冻结如何防止同一积分、停车券或使用次数被并发重复占用。
- 本屏保持业务状态机表达，不展开接口和数据库字段。

### S9 系统重新开始演进：真实问题进入内部闭环

使用四组「问题 → 优化」轨迹：

1. 权益扩展：会员等级、积分抵现、商户停车券、消费减免和便捷出场逐步进入统一体系。
2. 车辆风控：从仅限制会员升级为会员与车辆双维度限制，并增加绑定查询、后台解绑和操作日志。
3. 双重支付：针对 ETC 与小程序结果同步时间差，增加支付后 Loading 与等待出场提示。
4. 异常闭环：增加全额退款、现金与权益同步回退、失败步骤重试和跨日核销宽限。

本屏不平铺功能列表，而是证明问题能够被内部持续定位和处理。

### S10 真实落地：8 个 2.0 商场与 27 个存量商场

- 35 个停车商场中，8 个运行 2.0，27 个继续运行 1.0。
- 8 个 2.0 商场均使用完整缴费和权益能力。
- 8 个 2.0 商场约产生 7.57 万笔月停车缴费单。
- 某集团 19 个商场中，1 个使用 2.0、18 个使用 1.0。
- 存量迁移受到车辆、订单、规则和历史数据整合成本影响。
- 退款能力已上线但暂无真实订单。
- 视觉上分成 2.0 与 1.0 两条版本车道，表达渐进迁移，不将 1.0 车道画成失败或废弃状态。
- 不在本屏重复展示 S2 的停车业务背景金额。

### S11 结论：迁回的是产品定义权与演进能力

- 回顾从线上 1.0 黑盒还原能力、重新规划产品，到接回内部身份、权益、订单与外部停车系统。
- 强调不是重做页面、不是原样复制、不是全面替代。
- 收束到内部团队重新掌握停车产品的规划、维护和持续演进能力。
- 提供返回项目经历页的清晰入口。

## 9. 视觉系统

### 9.1 整体方向

采用「车道导视与闸机」视觉隐喻：

- 一条车道贯穿 11 屏，承担从 1.0 黑盒到 2.0 内部体系的叙事路径。
- 闸机开启表达 XCU 访客无需强制开卡即可缴费。
- 车道下方的分层空间承载内部业务、停车后端、停车中台和外部适配。
- S9 的问题进入循环维护区，S10 分成 2.0 与 1.0 两条版本车道。
- 视觉应保持抽象、克制和结构真实，避免车辆写实素材、速度线、镜头追车或汽车广告式氛围。

### 9.2 语义颜色

- 路面与背景：深石墨蓝，表达既有系统与复杂运行环境。
- 车道线与历史问题：琥珀色，表达待梳理边界、迁移提醒和风险。
- 2.0 原生体系与成功通路：青绿色，表达可解释、可维护和可继续演进。
- MCU 会员权益支线：紫色，区别于 XCU 访客直行支付。
- 外部适配与系统层：蓝色，表达协议和系统边界。
- 异常与阻断：玫红色，只用于支付失败、越限和需要回退的路径。

颜色必须同时配合标签、线型、箭头和文字，不作为唯一状态编码。浅色主题将路面转为浅灰蓝，不创建固定暗色孤岛。

### 9.3 关键图形

1. S1：从 1.0 黑盒驶向 2.0 内部体系的单条车道。
2. S2：三项业务规模卡与「到场—停车—消费—缴费离场」旅程。
3. S3–S4：黑盒与逐项照亮的页面、流程、规则、状态和异常清单。
4. S5：XCU 直行支付车道和可选择进入的 MCU 权益支线。
5. S6：内部业务、停车后端、停车中台与外部适配的纵向分层。
6. S7：四类权益汇入总减免额度，并显示会员 / 车辆双维限制。
7. S8：冻结、支付、扣减、解冻和通知的状态路径。
8. S9：问题进入内部迭代闭环。
9. S10：2.0 与 1.0 两条并存版本车道。

## 10. 动效与响应式设计

### 10.1 动效原则

- 不使用 GSAP 或 ScrollTrigger。
- 使用页面独立的 `IntersectionObserver + CSS` 动效控制器。
- 场景默认完整可见；脚本成功初始化后才添加动效准备状态。
- 每屏只进行一次短时入场、点亮、开闸或路径连接，不长时间锁定滚动。
- 动效只解释状态变化，不让车辆主题抢走产品叙事。
- 不修改 `DeckLayout` 的全局滚动行为。

### 10.2 重点动效

- S3–S4：从 1.0 黑盒依次照亮页面、流程、规则、状态和异常。
- S5：XCU 直行闸机打开，MCU 权益支线随后出现。
- S6：内部业务、停车后端、中台和适配层沿车道下方依次对齐。
- S7：四类权益按用户选择顺序进入总额度槽，超限路径停在边界。
- S8：冻结、支付成功、失败解冻和通知路径依次点亮。
- S9：真实问题进入内部修复闭环，不播放无限循环。
- S10：同一路径分成 2.0 与 1.0 两条并存车道。

### 10.3 移动端与减少动态

- 桌面端保持整屏阅读和车道空间关系。
- 移动端场景改为 `auto-height` 纵向排列，透视车道改为竖向路线。
- 权益、架构层和状态路径按阅读顺序堆叠，不出现横向溢出。
- `prefers-reduced-motion: reduce` 下关闭位移、开闸、路径流动和错峰动画，直接展示最终状态。
- 动效不能成为理解内容的前提。

## 11. 技术结构

### 11.1 页面职责

`src/pages/projects/smart-parking.astro` 负责：

- 11 屏语义化 HTML。
- 页面局部 CSS 和响应式规则。
- 脱敏后的 CSS/SVG 停车、身份、权益和交易图形。
- 章节导航与返回项目页入口。
- 经过核验的数据或无数据时的 S2 降级布局。

### 11.2 动效控制器职责

`src/scripts/smart-parking-motion.ts` 负责：

- 建立和释放 `IntersectionObserver`。
- 根据场景进入状态添加稳定的 `data-motion-state` 或类名。
- 识别桌面端、移动端和减少动态偏好。
- 页面切换或卸载时停止观察并清理监听。
- 初始化异常时回到静态可读状态。

动画脚本通过稳定的 `data-scene`、`data-motion` 和 `data-motion-state` 属性选择元素，不把视觉类名同时当作脚本接口。

### 11.3 范围隔离

- 不修改 `src/layouts/DeckLayout.astro`。
- 不修改销售线索、企业权限或其他项目 Deck 的内容、CSS、脚本和测试。
- `.superpowers/` 中的可视化伴侣草稿不进入正式提交。
- 第四个 Deck 的 Spec、页面、脚本和测试使用独立命名空间。

## 12. 降级与异常处理

| 情况 | 行为 |
| --- | --- |
| S2 真实数据未完成核验 | 不展示数字卡，使用顾客旅程和业务关系图，不使用示例数字 |
| JavaScript 或动效脚本加载失败 | 所有正文和图形保持最终静态状态，页面完整可读 |
| `IntersectionObserver` 不可用 | 不添加动效准备状态，保留静态内容 |
| 用户开启减少动态效果 | 关闭位移、开闸、路径流动和错峰，直接展示最终状态 |
| 移动端或窄屏 | 透视车道改为纵向路线，图形顺序堆叠，无横向溢出 |
| 深浅色切换 | 仅更新 CSS 变量，不重新初始化 observer |
| 长商场、厂商或权益标签 | 允许换行和容器增高，不通过缩小字号隐藏溢出 |
| 页面切换或卸载 | `disconnect()` observer，并清理媒体查询和事件监听 |

## 13. 测试与验收

### 13.1 内容与事实

- 路由存在，项目经历首页第四张案例卡可达。
- 页面包含 S1–S11 和 5 个章节导航项。
- S3 明确说明 1.0 仍能运行，但难以维护和继续演进。
- S4 明确说明无可用产品和技术文档，并通过线上后台与小程序完成黑盒还原。
- S5 明确说明 XCU—MCU 身份解耦属于 2.0 首发能力，访客无需强制开卡即可缴费。
- S6 的停车后端、中台和外部适配职责符合已确认架构。
- S7 完整包含会员等级、积分、停车券和消费减免，以及会员 / 车辆双维限制。
- S8 完整包含重新校验、冻结、支付、扣减 / 核销、解冻和外部通知。
- S9 包含车辆管理、双重支付、退款失败重试和跨日核销等真实迭代。
- S10 明确区分 8 个 2.0 商场与 27 个 1.0 商场，并说明退款暂无真实订单。
- 页面不出现客户名称、真实商场、车牌、订单、生产接口或未经核验的数据。
- 页面不出现可视化伴侣中的示例数字。

### 13.2 功能与交互

- 底部章节导航可到达对应场景。
- 主题切换后文本、车道、权益、状态和边界仍清晰可读。
- 场景进入时只执行局部短时动画，不出现长时间滚动锁定。
- 快速滚动和反向滚动不会留下半隐藏文字或不可恢复状态。
- 页面卸载后 observer 和媒体查询监听被清理。

### 13.3 响应式与无障碍

- 桌面端在 1440×900 和常见笔记本高度下不遮挡正文与底部导航。
- 移动端在 390×844 下按纵向顺序完整展示，无横向溢出。
- `prefers-reduced-motion: reduce` 下没有位移、开闸和路径流动动画。
- 键盘可聚焦返回链接、主题切换和章节导航。
- 颜色之外同时使用文字、线型、标签和图标表达状态。

### 13.4 数据闸门验收

- S2 每一项数字都能追溯到统计范围、指标定义和完整自然月。
- 如果 S2 使用全停车业务数据，明确说明是否包含 1.0 与 2.0。
- 如果 S2 使用代表性商场数据，明确说明商场范围而不外推为全平台。
- S2 不与 S10 重复展示同一组 2.0 月缴费单和应付金额。
- 如果数据未通过核验，生产页面使用无数字降级布局。

## 14. 设计结论

第四个 Deck 的内容与体验都围绕同一个判断组织：

> 将外包停车系统迁回内部的价值，不是拥有一份旧代码，而是重新掌握产品定义、内部业务连接和持续演进能力。

页面先以停车业务规模建立问题重要性，再用无文档黑盒还原证明项目难度；以 XCU—MCU 身份解耦、系统分层、权益编排和支付生命周期证明重构深度；最后用真实迭代、8 个 2.0 商场和 27 个存量商场共同呈现落地结果与迁移现实。
