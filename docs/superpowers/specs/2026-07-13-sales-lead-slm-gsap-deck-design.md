# 销售线索 SLM GSAP 试点 Deck 设计 Spec

> 日期：2026-07-13
>
> 状态：设计已确认，门外汉可读性迭代已完成
>
> 页面定位：个人网站「项目经历」首个旗舰案例，同时作为 GSAP 动效在本站的试点 Deck

## 1. 背景

个人网站项目经历模块已经确认四个案例大纲，但继续一次性讨论所有项目会增加沟通负担。因此，本阶段先独立启动销售线索 SLM Deck，不等待其他候选项目继续细化。

销售线索案例的价值不在功能数量，而在两次连续发生的产品判断：

1. 面对无法用真实数据验证的复杂门店评分模型，放弃伪精确的统一总分，改为可解释、可配置、可回收的动态匹配机制。
2. 第一阶段机制取得结果后，又从跨行业使用差异中识别出上游线索质量和责任错位，推动产品由“激励门店”转向“总部先承担基础质量责任，门店再主动付费认领”。

本 Deck 还承担一个受控试验：使用 GreenSock 官方 GSAP Skills 指导首屏入场、吸顶章节、横向滚动、轻视差和微交互设计。试点只作用于销售线索 Deck，完成体验、性能和可访问性验收后，才决定是否把相同能力复用到首页。

## 2. 已确认的核心决策

### 2.1 叙事路线

采用「产品演进与判断型」叙事：

```text
线索为什么持续流失
  -> 尝试用复杂评分寻找最优门店
  -> 识别伪精确，改为可解释的公海池机制
  -> 核销率提升，第一阶段得到验证
  -> 跨行业使用暴露适用边界
  -> 从激励门店转向重构总部、平台和门店的责任链
```

复杂系统设计是中段证据，商业数据是验证证据，二者都不取代产品判断主线。

### 2.2 读者记忆点

第一记忆点：作者不仅能设计复杂的分发机制，也能在真实数据暴露边界后重构整条责任链。

第二记忆点：作者从核心模块建设者成长为完整产品线负责人。

### 2.3 视觉方向

采用「经营系统蓝图」为主的视觉语言：

- 深色网格、路径、节点和同心扩圈表达系统编排。
- 强标题与克制留白表达关键判断。
- 大数字只用于已验证结果，不把整页做成经营后台。
- 已上线、已排期、规划中和开放验证使用清楚、稳定的状态色，不靠文案细节才能区分。

## 3. 目标与非目标

### 3.1 目标

- 让招聘者在一次完整滚动中理解销售线索产品如何从采集问题演进为经营责任系统。
- 清楚展示本人贡献边界、关键取舍、上线机制、业务结果和后续方向。
- 用约 10 个付费品牌、约 15,000 家上线门店和单客户结果数据证明真实落地。
- 通过 GSAP 试点验证滚动编排是否能提升复杂产品故事的理解效率。
- 保证动画关闭、加载失败、移动端和减少动态效果模式下仍可完整阅读。

### 3.2 非目标

- 不把销售线索写成作者从零独立建设的系统。
- 不展开 AI 外呼清洗和通话智能总结；AI 主题留在个人网站 AI 模块。
- 不公开客户、品牌、门店、顾客、手机号、地址、订单、线索编号或内部成员信息。
- 不展示生产环境截图、完整权限字典、接口、数据库字段或可被利用的内部规则。
- 不把清洗、线索点或付费公海池包装成已上线成果。
- 不在本试点中改造首页、其他 Deck 或公共滚动行为。
- 不引入 ScrollSmoother、React 动画封装或额外动画库。

## 4. 受众与成功标准

主要受众：评估作者 To B 产品能力、复杂系统设计能力和持续迭代能力的招聘者或业务负责人。

读完后应能回答：

- 为什么多平台采集只是表层问题，真正的问题是顾客意向在组织流转中衰减。
- 为什么复杂评分模型没有进入最终上线版本。
- 公海池如何同时处理位置保护、主动承接、扩圈和回收。
- 第一阶段为什么有效，以及为什么无法直接复制到所有行业。
- 清洗与付费认领分别重构了哪一段责任。
- 哪些内容已上线、已排期、规划中或仍需上线数据验证。

## 7. 逐屏内容设计

### S1 封面：让线索真正到达合适的门店

- 主标题：让线索真正到达合适的门店。
- 副标题：全域销售线索管理系统：从动态匹配到责任链重构。
- 规模证据：约 10 个付费品牌、约 15,000 家上线门店、5 类线索来源。
- 角色副线：从核心模块建设者到产品线负责人。
- 首屏不出现“从零建设”“独立完成整套系统”等越界表述。

### S2 真正的问题：流失的是顾客意向

- 先用六步业务链路建立背景：顾客留资或领券 → 平台产生线索 → 总部汇总线索 → 匹配合适门店 → 门店跟进顾客 → 核销或成交。
- 明示客户最初提出的是“多平台下载、导入和分配太麻烦”，这是采集与操作层的表象问题。
- 再说明继续追查后识别出的三个经营根因：缺少关键位置与门店信息、层层下载导入错过最佳跟进时间、无人跟进时缺少回收机制。
- 用“什么才是合适的门店”作为通往 S3 评分尝试的过渡问题。
- 明确早期接入底座由原产品负责人建设；作者从报表和分配问题进入，逐步主导后续经营链路。

### S3 第一次尝试：计算“最优门店”

- 线索数据：创建时间；有效性、价值和成交难度标签；地理位置及来源准确性。
- 门店数据：任务失败、跟进效率、成交绩效、客户意向度和意向门店成交绩效构成的门店评分；当前被分配和跟进中的线索负载；地理位置等基本属性。
- 方案曾考虑历史数据、近 180 天数据、线性公式、阶梯递减和离散分值。
- 本屏展示的是探索过程，不把这些模型写成已经上线的算法。

### S4 产品判断：放弃伪精确的统一总分

- 地理位置优先与服务能力优先产生直接冲突。
- 公平、翻盘机会、强店通吃、消极门店和恶意抢单也无法被一个未经验证的总分可靠表达。
- 团队没有足够数据和成本验证完整算法，继续计算只会制造虚假精确。
- 最终仅保留成交加分、超时回收扣分，以及经营分和门店负载的场景资格过滤。
- 核心判断：把评分从不可靠的预测工具降级为可解释的治理工具。

### S5 第一阶段上线机制：保护、扩圈、抢单与回收

- 第一次出现术语时用白话解释：公海池是无法直接分发给明确意向门店的线索，向符合资格的门店开放抢单；保护期是意向门店和最近门店先看、先抢，到期后再逐步扩圈。
- 能通过线索初分配直接命中的线索生成门店任务，不进入公海池。
- 意向门店和购买线索时最近的门店最早获得公海池可见与抢单资格。
- 场景开始时间和阶段保护时间共同控制扩圈节奏。
- 保护期结束后，3 公里、5 公里、10 公里、30 公里等范围逐级加入；最早获得资格的门店仍然可见，不因扩圈被移出。
- 场景先校验经营分、门店负载和其他资格条件，再开放抢单。
- 一家门店抢单成功后，线索离开公海池并生成任务。
- 超时任务回收后重新进入智能分发，曾经被分配或抢单过的门店被过滤，避免循环回到相同承接者。

### S6 第一阶段结果：主动承接得到验证

- 同一客户、同一统计口径。
- 观察期约 3 个月，月均约 10 万条线索。
- 用完整业务口径说明：在这个客户的业务里，卡券产生的销售线索完成核销后计为已成交。
- 卡券核销率由不足 49.4% 提升至 61.7%，绝对提升超过 12.3 个百分点。
- 本屏不把结果外推到所有品牌，也不声称仅由单一功能产生严格实验因果。

### S7 跨行业反转：成功条件也是适用边界

- 头部家电客户拥有更强的平台流量扶持和广告预算，线索意向高、成交链路短、门店密度高；部分顾客甚至已在门店购买卡券。
- 小众品牌、全屋定制和家具客户预算更有限、成交链路更长，一个城市可能只有一两家门店。
- 在低门店密度场景中，地理位置错误会被放大，线索更容易进入完全不合适的承接范围。

### S8 根因重判：下游激励不能修复上游真实性

- 总部承担投流费用，门店承担联系和跟进成本。
- 当手机号非本人、顾客只是误触客服或表单、地理位置错误时，经营分、抢单和回收都无法修复线索本身。
- 若继续加重门店处罚，等于让门店为上游数据错误承担责任。
- 核心判断：要求门店负责之前，系统必须先证明线索具备被跟进的基本资格。

### S9 已排期：清洗层重建基础质量责任

- 确认手机号对应本人。
- 确认地理位置和最近门店。
- 确认初步意向，即顾客允许稍后由专业人员继续跟进。
- 清洗过程中可补充顾客确认的意向门店。
- 清洗完成后选择对应场景，进入“直接分发给意向门店”或“进入付费公海池”两条路径。
- 清洗只承担基础真实性责任，不承诺成交，不在项目经历 Deck 展开 AI 清洗实现。

### S10 方案规划中：付费认领重建承接意愿

- 门店线下完成充值，在后台提交充值工单。
- 后台管理人员审核后向对应门店增加线索点。
- 门店进入付费公海池抢单，抢单成功后获得线索并消耗线索点。
- 系统暂不经手真实资金，避免在当前阶段引入支付、结算和退款体系。
- 本屏必须显著标注“方案规划中，尚未完成详细 PRD”，不展示虚构的上线结果。

### S11 开放结尾：如何证明责任重构成立

- 下一阶段验证指标：有效线索率、门店认领率、首次跟进时长、质量申诉率、认领后成交率。
- 公开说明付费认领后的质量申诉和点数返还仍需产品决策。
- 用“从核心模块建设者到产品线负责人”收束个人角色成长。
- 结尾停在可验证的问题上，不使用庆祝或完成态动效。

## 8. GSAP 动效试点设计

### 8.1 总体原则

- 动效只解释状态变化、因果关系和空间扩展，不给每个元素增加无意义运动。
- 使用两段核心滚动编排，其余场景只做一次性、短时入场。
- 首屏入场、吸顶章节、横向滚动、轻视差和微交互各自只在最适合的内容位置出现。
- 所有核心内容先以可读 HTML 存在；GSAP 只增强呈现，不承担内容是否可见的唯一责任。

### 8.2 首屏入场：S1–S2

- 使用一个 `gsap.timeline()` 组织标题、来源平台节点、连接路线和门店节点。
- 总时长控制在约 1.2 秒，使用 `autoAlpha`、`x`、`y` 和 `stagger`。
- 背景网格只允许轻微视差，不能影响文字清晰度。
- S2 进入时依次显现六步业务链路；客户表层诉求与三个经营根因保持静态可读，不依赖动画才能理解。

### 8.3 核心编排一：S3–S5

- 使用桌面端吸顶章节，滚动驱动一个顶层 timeline。
- 第一阶段展开线索三类数据与门店三类数据。
- 第二阶段让距离、服务、公平和反作弊之间的冲突变得可见。
- 第三阶段让复杂总分连线失效，并重组成经营分过滤、负载过滤、保护期、分级扩圈和回收机制。
- ScrollTrigger 绑定在顶层 timeline，不在子 tween 上重复创建。
- 被 pin 的是章节舞台，动画作用于舞台内部子元素，不直接动画 pinned 元素。

### 8.4 结果屏：S6

- 将不足 49.4% 平稳过渡到 61.7%，同时依次出现观察期、月均线索量和成交定义。
- 数字动画只播放一次，完成后停止，不循环、不弹跳。
- 不使用视觉上暗示严格 A/B 实验或全量客户共同结果的图表。

### 8.5 核心编排二：S7–S8

- 桌面端采用垂直滚动驱动的横向内容轨道。
- 三个面板依次展示：头部家电成功条件、长链路行业限制、总部与门店责任错位。
- 横向 tween 使用 `ease: "none"`，确保滚动位置与内容位置一一对应。
- pinned 容器与横向移动的内部 track 分离。
- 本段不使用嵌套 ScrollTrigger；如果没有对子面板单独触发动画的必要，不引入 `containerAnimation`。

### 8.6 责任重构：S9–S11

- 清洗节点、意向门店补全、场景选择、充值工单和抢单扣点沿一条责任链依次接通。
- 已排期、方案规划中、开放验证使用不同状态色。
- 普通节点使用同一章节 timeline 的 `stagger` 一次性显现，不为每个节点单独创建 ScrollTrigger，也不再增加第三段大型 pin/scrub 编排。
- 结尾停在验证指标与开放问题，不产生“已经成功”的完成态动画。

### 8.7 微交互

- 章节导航随当前章节更新激活状态。
- 案例卡、状态标签和机制节点只允许轻微位移、边框或光晕变化。
- 返回按钮、主题按钮和项目入口保留清晰焦点态。
- 不增加鼠标跟随、持续旋转、无限呼吸或与内容无关的粒子背景。

## 9. 技术设计

### 9.1 运行时依赖

- 从公开 `gsap` npm 包引入 `gsap` 和 `ScrollTrigger`。
- 只注册 `ScrollTrigger`，不引入 ScrollSmoother 或其他高级插件。
- 官方 GSAP Agent Skills 已安装到共享目录，用于约束实现与审查；它们不是网站运行时依赖。

### 9.2 页面结构

- `src/pages/projects/sales-lead-slm.astro` 负责语义化内容、SVG/CSS 示意图和页面局部样式。
- `src/scripts/sales-lead-slm-motion.ts` 负责 GSAP 注册、媒体条件、timeline 创建、ScrollTrigger 创建与清理。
- 页面使用脱敏后的 CSS/SVG 业务示意，不加载生产截图。
- 动效脚本通过稳定的 `data-motion`、`data-scene` 和 `data-state` 属性选择元素，避免把视觉类名同时当脚本接口。

### 9.3 与 DeckLayout 的滚动关系

现有 `DeckLayout` 默认让 `main` 成为 `100vh` 内部滚动容器，并启用 `scroll-snap-type: y mandatory`。强制吸附会与 GSAP 的连续 `scrub`、`pin` 和横向轨道争夺滚动位置。

销售线索页面单独覆盖为连续原生页面滚动：

- `.sales-lead-deck` 使用 `height: auto`、`overflow: visible`、`scroll-snap-type: none`。
- `.sales-lead-deck section` 使用 `height: auto` 和 `min-height: 100svh`，不继承固定整屏高度与强制吸附。
- 内容场景使用 `min-height`，不强制每个场景固定为完整视口。
- 两个 pinned 章节由 ScrollTrigger 生成滚动空间。
- 覆盖规则只作用于 `.sales-lead-deck`，不修改 `DeckLayout` 的全局默认行为，不影响现有 Deck。

### 9.4 初始化与清理

- 页面加载后注册一次 `ScrollTrigger`。
- 使用 `gsap.matchMedia()` 建立桌面、移动端和减少动态效果三套条件。
- 以 `gsap.matchMedia()` 作为响应式动画的生命周期容器，不在其回调内部再嵌套 `gsap.context()`；保存 matchMedia 实例，离开页面或发生 Astro 页面切换时统一 `revert()`，并清理剩余的顶层 timeline / ScrollTrigger。
- 字体和影响布局的资源完成后调用一次 `ScrollTrigger.refresh()`。
- ScrollTrigger 按页面从上到下的顺序创建；异步布局变化不得随机重建触发器。

### 9.5 响应式与减少动态效果

- 桌面端启用两段 pin/scrub 和一段横向轨道。
- 移动端取消 pin 和横向移动，所有内容恢复纵向自然顺序，仅保留轻微一次性显现。
- `prefers-reduced-motion: reduce` 下跳过入场、scrub、pin、横向移动和数字过渡，直接展示最终状态。
- 页面不能依赖 JS 设置初始不可见状态；JS 失败时正文、数字和流程必须可见。

### 9.6 性能边界

- 优先动画 `x`、`y`、`scale`、`rotation` 和 `autoAlpha`。
- 不用 `width`、`height`、`top`、`left`、`margin` 或 `padding` 驱动主要运动。
- `will-change` 只添加给真正参与动画的元素，并在长期不需要时清除。
- 同一时间只运行当前章节的动画，避免大量离屏 tween 持续工作。
- 试点目标为不超过 8 个活跃 ScrollTrigger，其中只有 2 个大型 pinned timeline。
- 生产环境不得保留 ScrollTrigger markers。

## 10. 降级与异常处理

| 情况 | 行为 |
| --- | --- |
| GSAP 包或页面脚本加载失败 | 页面保持完整静态 HTML，所有正文、结果和状态可见 |
| 用户开启减少动态效果 | 跳过所有大型动效，保持纵向静态阅读 |
| 移动端或窄屏 | 取消吸顶和横向轨道，按 S1–S11 顺序纵向展示 |
| 字体或资源导致布局变化 | 资源稳定后执行一次 `ScrollTrigger.refresh()` |
| 页面切换或组件卸载 | 统一清理 timeline、matchMedia 和 ScrollTrigger，避免残留监听与 pin spacer |
| 用户快速滚动或反向滚动 | scrub 动画跟随当前进度，不依赖只能向前播放的隐藏状态 |
| 深浅色切换 | 只切换 CSS 变量，不重建动画结构；必要时在布局未变化时不调用 refresh |

## 13. 风险与控制

### 风险一：动效成为主角

控制方式：只保留两段核心滚动编排，每段都必须对应一个产品判断；删除无法解释内容变化的动画。

### 风险二：scroll-snap 与 ScrollTrigger 冲突

控制方式：销售线索页面局部关闭 scroll-snap，使用连续页面滚动；其他 Deck 保持原样。

### 风险三：未来方案被误解为已上线

控制方式：状态标签常驻；S9、S10、S11 使用不同视觉状态，结果屏只承载已验证数据。

### 风险四：移动端 pin 和横向滚动不可用

控制方式：移动端不做同构动画，直接采用纵向内容结构。

### 风险五：试点被过早复制到首页

控制方式：本次不改首页。只有在内容理解、移动端体验、减少动态效果、性能和维护成本全部通过验收后，才另开设计决策判断首页复用范围。

## 14. 完成定义与后续决策

本 Deck 的设计目标在以下条件同时满足时才算完成：

- 11 屏内容和事实口径全部落地。
- 两段核心 GSAP 编排能够解释对应的产品判断。
- 静态、移动端和减少动态效果模式都能完整阅读。
- 本地检查、构建和相关 Playwright 验收通过。
- 项目经历首页存在可用入口，但首页未发生变更。

完成后另行评估首页复用，不自动扩大范围。评估问题包括：首屏入场是否提升理解、pin/horizontal 是否造成阅读阻力、移动端降级是否自然、维护成本是否可接受，以及动效是否仍然服务内容而非品牌炫技。
