# 多平台会员运营体系 Deck 设计 Spec

> 日期：2026-07-18
>
> 状态：设计已确认，等待用户复核书面 Spec
>
> 页面定位：个人网站「项目经历」第五个公开案例；以继承的会员身份底座为起点，展示多平台扩展、增长归因与风险治理的体系演进

## 1. 背景

个人网站前两个已实现的项目经历 Deck 分别证明了两类产品能力：销售线索 SLM 展示产品演进、复杂机制编排和责任链重构；多业务线企业权限体系展示共享底座上的差异建模与组织治理。第五个公开案例需要提供不同的能力证据：在继承既有会员身份模型的前提下，如何把分散的平台身份、交易兼容、会员触达、渠道归因和异常治理逐步连接为一套可持续运营的 CRM 会员体系。

本案例不能从 MCU、ECU、XCU 等内部术语直接开场。陌生访客需要先理解：CRM 管理的不是某一次页面访问，而是同一会员长期累积的会员卡、积分、卡券、消费、等级和互动关系。随着业务从微信、支付宝扩展到抖音、小红书、快手和线下二维码，同一顾客会以多个平台访客身份进入系统，触达、交易、归因和治理也会随之断开。身份模型是解决这些问题的基础，但不是案例的最终主角。

本人的贡献起点也必须准确呈现。早期 MCU—ECU—微信 / 支付宝身份模型由其他产品经理规划；本人接手后理解并延续这一设计，主导后续平台接入、身份溯源适配、无手机号交易兼容、会员运营公共组件、推广归因和异常会员治理。案例证明的是在既有底座上持续扩展和形成闭环的能力，不认领原始身份模型的从零设计成果。

## 2. 已确认的核心决策

### 2.1 第一记忆点

访客看完后应首先记住：

> 作者能在继承既有身份底座后，把分散的平台、触达、归因和风控能力演进成一条可追踪、可运营、可处置的会员业务闭环。

跨平台身份重绑定和券主 MCU 是前半段的关键产品取舍；异常会员治理是后半段的产品判断高潮。它们共同服务体系演进主线，不将案例拆成多个孤立功能专题。

### 2.2 叙事路线

采用「体系演进型」叙事：

```text
会员为什么是 CRM 的核心
  -> 多平台扩张让身份和业务关系断开
  -> 继承 MCU—ECU—XCU 身份底座
  -> 完成平台访客身份重绑定
  -> 用券主 MCU 兼容无手机号交易
  -> 建设可复用的会员触达基建
  -> 用推广渠道与二维码形成行为归因
  -> 真实运行数据证明体系进入业务
  -> 店员刷单演变为跨店互刷与黄牛刷单
  -> 从身份硬规则升级为可预警、可核查、可纠错、可处置的治理闭环
```

### 2.3 开篇原则

- 第一章先解释会员对于 CRM 的业务意义，再进入具体身份模型。
- 需求背景来自真实业务扩张：平台、内容渠道、交易渠道和线下点位增加，同一会员以多个访客身份进入系统。
- MCU、ECU、XCU 延后到第四屏首次集中解释，且必须同时使用白话业务定义。
- 不把开篇写成 CRM 行业教程，只保留理解本案例所需的最小背景。

### 2.4 券主 MCU 的位置

券主 MCU 单独占一屏，作为身份扩展章节的第二个关键取舍：

- 上游不提供手机号时，系统仍需保障订单、卡券投放和核销链路连续。
- 能通过已有 XCU / ECU 找到真实 MCU 时，优先复用已有会员。
- 无法找到时，使用平台会员标识实例化独立券主 MCU 和 ECU。
- 券主 MCU 可能与真实 MCU 长期并存，不能包装成身份问题已经被完美解决。

### 2.5 产品判断高潮

全篇唯一高潮是异常会员治理：

> 真正的治理不是更重的封禁，而是把风险处置变成可发现、可核查、可纠错、可处置的闭环。

客户最初通过“本店员工订单不产生积分、不计会员升级”限制店员刷单，真实对抗随后转向跨店互刷和黄牛刷单，证明单一身份硬规则可以被绕过。系统因此升级为可配置异常预警、人工核查、白名单与禁用策略。禁用只限制会员升级和积分消耗，仍允许获取积分、自然降级和领取免费券，体现避免误伤和保留基础触达的边界判断。

### 2.6 结果证明

结果屏使用「平台采用规模 + 单客户触达覆盖」证明真实运行：

- 排除模拟商户后，平台共管理 8,764,906 条 MCU 记录。
- MCU 是客户主体下的会员记录数，不是跨客户去重后的独立自然人数。
- 某匿名客户一年新增开卡会员 113,722 人，其中开卡后营销计划成功发送 100,829 人，触达覆盖率约 88.7%。
- 该计划发生在开卡之后，不能反向表述为短信带来 11.37 万人开卡。

年度积分和销售额只用于说明体系承载的经营规模，不作为该营销计划的增量结果。首版 Deck 不将它们设为主结果数字，避免访客误读因果关系。

## 3. 目标与非目标

### 3.1 目标

- 让陌生访客理解会员为什么是 CRM 中连接资产、交易、互动和治理的主线。
- 准确展示本人继承原始身份模型并继续扩展的贡献边界。
- 用身份重绑定和券主 MCU 证明对身份归属、资产边界和业务连续性的判断。
- 用人群、短信、短链、推广渠道和二维码展示可复用的增长基建与完整归因链路。
- 用真实刷单对抗和有限禁用策略展示风险治理能力。
- 用约 876 万条 MCU 记录和约 88.7% 开卡后触达覆盖证明体系已真实运行。
- 与销售线索和权限体系 Deck 保持明显的叙事与视觉差异。

### 3.2 非目标

- 不写成完整 CRM、CDP 或会员中台产品说明。
- 不把原始 MCU—ECU—微信 / 支付宝身份模型写成本人原创。
- 不把身份重绑定写成自动识别同一自然人或合并全部会员资产。
- 不把券主 MCU 写成已经消除全部身份重复与数据债务。
- 不把顾客列表、人群筛选、短信、短链和二维码写成功能清单。
- 不把约 876 万条 MCU 写成 876 万独立用户。
- 不把开卡后触达覆盖率写成拉新转化率。
- 不把年度积分与销售规模包装成营销计划带来的增量结果。
- 不公开具体风控阈值、生产规则组合或真实异常会员。
- 不使用生产后台截图、真实客户名称、会员资料、手机号、订单或活动数据。
- 不使用大型吸顶、滚动 scrub、横向轨道或新增动画运行时依赖。
- 不修改已有 Deck 的内容、动效或公共布局行为。

## 4. 受众与成功标准

### 4.1 主要受众

- 招聘方和业务负责人：判断候选人是否能处理复杂 CRM 会员体系和长期运营问题。
- 产品负责人：判断身份边界、业务连续性、增长归因与风险治理中的产品取舍。
- 研发与数据协作者：判断方案是否尊重身份归属、会员资产边界、上游数据限制和公开口径。

### 4.2 内容成功标准

- 访客能解释会员为什么是本案例中的 CRM 主线。
- 访客能区分 MCU、ECU、XCU 的作用，但不需要理解内部数据库实现。
- 访客能复述“重绑定迁移的是当前平台身份归属，不是旧会员资产”。
- 访客能解释券主 MCU 为什么保障交易连续，以及为什么它仍是技术债。
- 访客能复述完整的会员触达与推广归因链路。
- 访客能理解异常治理为何从身份硬规则升级为预警、核查和有限处置。
- 访客不会误解本人贡献边界、用户规模或营销效果。

### 4.3 体验成功标准

- 12 个场景按四章纵向完整可读。
- 一条持续的“会员主线”帮助访客理解同一关系如何从身份扩展到运营和治理。
- 页面不复刻销售线索 Deck 的大型滚动编排，也不复刻权限体系 Deck 的双业务对照与矩阵语言。
- 桌面端、移动端、减少动态和脚本失败状态均不丢失语义。
- 深浅色主题下，颜色、标签、线型和正文共同表达状态，颜色不是唯一编码。

## 7. 逐屏内容设计

### S1 封面：把分散的平台访客，变成可持续经营的会员关系

- 主标题先表达业务价值，不使用 MCU、ECU、XCU。
- 副标题：多平台会员运营体系——从身份扩展到增长与风控闭环。
- 可使用三项范围标签：多平台身份、会员运营、异常治理。
- 首屏不使用“从零建设”“统一自然人”或“876 万用户”等越界表述。

### S2 会员为什么是 CRM 的主线

- CRM 管理的不是一次访问，而是会员持续累积的卡、积分、卡券、消费、等级和互动关系。
- 同一会员在不同时间、平台和业务中发生的行为需要被持续连接，运营与服务才能建立上下文。
- 使用一条“会员关系主线”连接资产、交易、互动和治理，不使用内部实体图作为主视觉。

### S3 需求背景：业务扩张把同一会员拆成多个访客

- 早期平台主要是微信和支付宝小程序。
- 后续扩展到抖音、小红书、快手以及线下二维码等内容、交易和点位入口。
- 同一顾客可能以多个 XCU 进入系统；如果身份、触达、交易和来源彼此断开，CRM 只能看到零散访问，无法形成持续经营关系。
- 本屏用业务问题引出身份底座，不声称系统能够跨客户识别同一自然人。

### S4 接手边界：继承 MCU—ECU—XCU 身份底座

- MCU：客户主体下承载会员卡、积分、卡券、消费和等级等资产的会员身份。
- XCU：顾客在某一小程序平台中的访客身份。
- ECU：连接 MCU 与不同平台 XCU 的关系层。
- 原始微信 / 支付宝模型由其他产品经理规划；本人从理解、吸收和后续扩展开始。
- 使用简化三层关系图，不公开数据库字段或内部接口。

### S5 关键取舍一：重绑定平台身份，不合并旧会员资产

- 同一平台 XCU 使用新手机号合法开卡并命中 MCU-B 后，相关平台身份迁移到 ECU-B。
- ECU-B 关联 MCU-B，成为当前平台访问与业务关系的归属。
- MCU-A 的积分、卡券、消费和等级不迁移；ECU-A 和 MCU-A 的历史关系继续保留。
- 视觉上区分“当前归属迁移”和“资产保持不动”，避免箭头暗示账户合并。
- 结论：平台访客身份溯源与重绑定，不是自动识别自然人或合并全部会员资产。

### S6 关键取舍二：券主 MCU 保证无手机号交易不断链

- 部分外部平台订单只返回平台会员标识，不返回手机号。
- 能通过已有 XCU / ECU 找到 MCU 时，优先复用已有会员。
- 无法找到时，以平台会员标识实例化独立券主 MCU 和 ECU，使订单、投放记录和核销继续运行。
- 由于上游始终缺少手机号，券主 MCU 可能与真实 MCU 长期并存，不能可靠自动合并。
- 本屏同时展示“业务连续性收益”和“身份债务边界”。

### S7 从身份底座到可复用的会员运营基建

- 顾客列表以会员身份汇总资料、平台访问、开卡、积分、卡券和订单。
- 人群筛选器与定向人群包作为所有需要会员范围的业务模块的公共组件。
- 公共短信服务统一管理模板、计划、执行记录和回调，避免各模块重复建设。
- 本屏只解释公共能力如何复用，不罗列完整后台字段和页面。

### S8 一条可审核、可执行、可追踪的触达链路

```text
筛选会员 / 创建定向人群包
  -> 创建短信计划
  -> 内容与额度审核
  -> 立即或定时发送
  -> 顾客点击短链
  -> 访问指定小程序页面
```

- 短信审核用于控制内容合规、额度和骚扰风险。
- 审核员可以指定成员，也可以按角色动态匹配。
- 短链携带商场、目标页面和推广渠道参数。
- 本屏强调触达链路是可运营流程，不是单次短信发送按钮。

### S9 从线上短链到线下二维码的完整渠道归因

```text
推广渠道 / 渠道明细
  -> 短链或线下二维码
  -> 小程序访问
  -> 开卡 / 领券 / 核销 / 下单
  -> MCU 串联会员行为
  -> 推广渠道串联本次来源
```

- 短链点击与小程序访问使用同一页面访问日志，日志携带推广渠道。
- 线下二维码可绑定停车场、电梯、门店或具体点位的渠道明细。
- 开卡、领券、核销和下单继续关联 MCU 与推广渠道。
- 完整链路已上线，但是否产生增量仍取决于活动设计和对照口径。

### S10 真实运行证据：规模与开卡后触达覆盖

- 平台管理约 876 万条 MCU 记录，明确标注“客户主体下的记录数，不是独立自然人数”。
- 匿名客户一年新增开卡会员约 11.37 万人。
- 其中开卡后营销计划成功发送约 10.08 万人，触达覆盖率约 88.7%。
- 数据卡片使用静态结果板，不进行庆祝式滚动数字。
- 不将触达覆盖率写成拉新转化，不将年度积分和销售规模写成营销增量。

### S11 真实对抗：单一身份硬规则被跨店互刷绕过

```text
本店员工刷单获取积分和会员等级
  -> 本店员工订单不产生积分、不计会员升级
  -> 店员转向跨店互刷，品牌门店出现黄牛刷单
  -> 单一身份硬规则失效
```

- 本屏承担从增长闭环切换到风险治理的转折。
- 不公开具体阈值、名单或可被利用的风控组合。
- 核心问题不是“规则写得不够多”，而是静态身份标签无法覆盖变化中的异常行为。

### S12 产品判断高潮：把封禁升级为有边界的治理闭环

- 客户可按全部门店、分类、楼层、标签、业态或指定门店配置规则范围。
- 规则可识别单笔高金额、单店高频 / 高累计消费和跨店高频 / 高累计消费。
- 系统每日计算异常会员和触发订单，并通知运营人员核查。
- 运营可将正常高频消费者加入白名单，也可禁用确认存在风险的会员。
- 禁用只限制会员升级和积分消耗，仍允许获取积分、自然降级和领取免费券。
- 最终结论：身份不是终点，它让每次触达、响应、交易和处置回到同一条会员业务主线。
- 页面提供返回项目经历页的清晰入口。

## 8. 视觉系统

### 8.1 整体风格

- 延续 `DeckLayout` 的极光背景、网格、玻璃面板和站点深浅色主题。

- 一条纵向会员主线贯穿四章；平台身份汇入主线，运营行为沿主线展开，风险治理在主线末端闭环。

## 9. 动效与响应式设计

### 9.1 动效方向

采用「引导式会员主线」：

- 页面保持正常纵向阅读，不锁定用户滚动。
- 场景进入视口时，会员主线推进到当前节点。
- 每屏只使用一次短时入场、连线点亮或状态切换。
- 不使用大型 pin、scrub、横向滚动轨道或持续循环动画。
- S5 重绑定、S8 触达链路和 S11 风险升级使用局部状态变化解释因果。

该方向使用与权限 Deck 相同级别的原生观察能力，但视觉语言和场景状态不同：权限 Deck 强调两套模型对照和维度映射；本 Deck 强调同一会员关系持续生长。

### 9.2 运行方式

- 使用页面独立的 `IntersectionObserver + CSS` 动效控制器。
- 场景默认保持最终静态可见；脚本成功初始化后才进入动效准备状态。
- 动效控制器更新场景可见状态、当前章节和会员主线节点。
- 不修改 `DeckLayout` 的全局 scroll-snap 行为。
- 不新增 GSAP timeline，也不增加其他动画库。

### 9.3 移动端与减少动态

- 桌面端保持整屏导航和左右 / 上下组合布局。
- 移动端场景改为自然高度纵向排列，会员主线移到内容卡片左侧。
- 平台身份、触达步骤、渠道归因和治理流程按 DOM 顺序堆叠，不产生横向溢出。
- `prefers-reduced-motion: reduce` 下关闭位移、错峰、连线推进和状态过渡，直接展示最终状态。
- 动效不能成为理解身份迁移、触达链路或治理结果的前提。

## 10. 技术结构

### 10.1 页面职责

`src/pages/projects/membership-operations.astro` 负责：

- 12 屏语义化 HTML。
- 页面局部 CSS、主题变量扩展和响应式规则。
- 脱敏后的 CSS / SVG 会员关系图形。
- 四章导航、状态标签和返回项目页入口。
- 页面级描述、标题和可访问性文本。

### 10.2 动效控制器职责

`src/scripts/membership-operations-motion.ts` 负责：

- 建立和释放 `IntersectionObserver`。
- 根据场景进入状态更新稳定的 `data-motion-state`。
- 更新当前章节导航和会员主线当前节点。
- 识别桌面端、移动端和减少动态偏好。
- 页面切换或卸载时停止观察并清理媒体查询监听。
- 初始化异常时切回静态可读状态。

动效脚本通过稳定的 `data-scene`、`data-chapter`、`data-member-node` 和 `data-motion-state` 属性选择元素，不将视觉类名同时作为脚本接口。

### 10.3 范围隔离

- 不修改 `src/layouts/DeckLayout.astro`。
- 不修改销售线索和企业权限 Deck 的页面、脚本、样式或事实口径。
- 不修改现有全局主题和滚动行为。
- `.superpowers/` 可视化草稿不进入正式提交。
- 实现范围只包含会员运营 Deck、项目经历入口和对应测试。

## 11. 降级与异常处理

| 情况 | 行为 |
| --- | --- |
| JavaScript 或动效脚本加载失败 | 所有正文、图形、数据与状态保持最终静态可见 |
| `IntersectionObserver` 不可用 | 不添加动效准备状态，直接使用静态模式 |
| 用户开启减少动态效果 | 关闭位移、错峰、连线推进和状态切换 |
| 移动端或窄屏 | 取消固定整屏高度，图形按 DOM 顺序纵向重排 |
| 深浅色切换 | 只更新 CSS 变量，不重建 observer |
| 长平台名、渠道名或术语解释 | 允许换行和容器增高，不通过缩小字号隐藏内容 |
| 页面切换或卸载 | `disconnect()` observer，并清理媒体查询与页面事件监听 |
| 初始化过程抛出异常 | 标记 fallback 模式，恢复全部静态内容并保留可诊断日志 |

## 14. 设计确认结论

本 Deck 最终采用：

- 四章、12 屏的体系演进叙事。
- 先讲会员与 CRM 的业务背景，再进入身份模型。
- 券主 MCU 单独成屏。
- 异常会员治理作为全篇产品判断高潮。
- 「会员主线」作为全篇视觉隐喻，中后段局部使用增长 / 治理双环。
- 引导式纵向动效，不使用大型滚动编排。
- 所有数据、身份与风控口径遵守项目总纲中的公开边界。

该设计范围足够集中，可在用户书面复核后进入单一实施计划。
