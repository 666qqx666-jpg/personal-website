# Skill Desk 设计 Spec

## 1. 背景

个人网站的 AI 落地实战栏目已经形成三条内容线：

- `Personal Knowledge Harness`：讲个人知识库、三层记忆和多 agent 共享上下文的完整系统。
- `从 Claude Code 到 AI 产品架构`：讲从 coding agent 源码中抽象出的 AI 产品架构方法。
- `置身钉内：责任流阅读笔记`：讲一次深度阅读如何产出 To B / 企业 AI / 责任流判断。

AI 栏目中原本还有一个未落页的 `reading-dialogue Skill` 卡片。它不应再作为孤立占位存在，而应升级成一个更长期的内容入口：`Skill Desk`。

`Skill Desk` 用来展示作者把日常高频 AI workflow 设计成可复用 skill 的过程。它不是普通工具目录，也不是只给自己看的说明书，而是一个兼具作品集展示和自用工作台的 skill 方法库。

本 spec 的直接来源包括：

- 当前对话确认的方向：`A + B 先落地，C 作为每个 skill 的产品化判断模块`。
- 来源线程 `019f3238-3ca3-7d91-9237-7cb4273f229a` 中确认的旗舰子页主线：`reading-dialogue Skill：把深度阅读变成知识生产`。
- 现有 shared skill：`/Users/qqx/.agents/skills/reading-dialogue/SKILL.md`。
- Obsidian 规则：知识卡片入库闸门、三层知识库、正式知识域按需加载、系统线头看板。

## 2. 定位

页面总标题：

`Skill Desk`

副标题：

把实用 AI workflow 拆成可解释、可验证、可继续产品化的 skill 方法库。

一句话定位：

`Skill Desk` 展示的不是“我会用哪些 AI 工具”，而是我如何把反复出现的工作场景拆成有边界、有输入输出、有失败复盘、有产品化潜力的 agent skill。

第一版定位采用：

- **B 为主：自用工作台**  
  让作者能回看每个 skill 的设计思路、真实问题、工作流、挫折和后续优化方向。
- **A 做入口：作品集展示**  
  让访客快速看懂作者具备 AI workflow / agent 产品设计能力。
- **C 做收束：可产品化判断**  
  每个 skill 子页保留“企业级复用 / 开源 GitHub 项目”的判断模块，但第一版不把它做成完整产品孵化系统。

## 3. 受众与阅读目标

主要受众：

- 评估作者 AI 产品能力、agent workflow 设计能力的访客。
- 对个人知识库、多 agent、AI 工作流、skill 化方法感兴趣的人。
- 作者自己，作为后续复盘、迭代和产品化候选管理入口。

读完后希望访客记住：

- 作者不是只会使用现成 AI 工具，而是在把重复工作抽象成可复用 agent skill。
- 每个 skill 都来自真实使用、真实失控、真实修正，而不是一次性写出的 prompt。
- 作者看 AI 产品时，会关注输入、上下文、状态、边界、失败、验证和可复用路径。

## 5. 首页设计

`/ai/skill-desk/` 是一个可扩展目录页，不做营销 landing，也不做单纯卡片堆叠。首页第一屏就应该让用户看到 skill 列表和筛选结构。

### 5.1 首页目标

首页回答四个问题：

1. 这里收录了哪些 skill？
2. 每个 skill 解决什么真实工作问题？
3. 它当前成熟到什么程度？
4. 它有没有变成团队模板、开源项目或企业级工具的潜力？

### 5.2 首页结构

建议结构：

1. 顶部标题区  
   标题 `Skill Desk`，副标题一行，下面放 3 个轻量统计：
   - `已稳定使用`
   - `迭代中`
   - `可产品化候选`

2. 场景筛选区  
   使用分段控件或 tabs：
   - `全部`
   - `阅读与沉淀`
   - `复盘`
   - `PRD / Spec`
   - `研究与分析`
   - `产品化候选`

3. Skill 卡片网格  
   每张卡片展示：
   - skill 名称
   - 一句话问题
   - 使用场景
   - 成熟度标签
   - 输出物标签
   - 产品化潜力标签

4. 当前旗舰区域  
   首页第一版需要突出 `reading-dialogue Skill`，因为它已有完整真实案例、失控复盘和优化链路。

5. 产品化候选区  
   展示哪些 skill 未来可能变成：
   - GitHub 开源项目
   - 团队内部 SOP
   - 企业知识工作流模板
   - 可视化工作台

## 6. 子页通用模板

每个 skill 子页使用统一结构，便于未来批量扩展。

### 6.1 子页固定模块

1. **封面**  
   标题、定位、一句话问题。

2. **真实问题**  
   这个 skill 从什么真实痛点里长出来。

3. **为什么不是普通 prompt**  
   说明它为什么需要工作流、状态、边界、输入输出和验证。

4. **设计思路**  
   拆解核心原则、使用边界、上下文策略。

5. **具体实现流程**  
   用 pipeline 图或步骤图呈现。

6. **事故现场 / 挫折**  
   展示一次真实失控或失败，以及它暴露的系统问题。

7. **优化机制**  
   说明后来增加了哪些规则、刹车、门禁或索引维护。

8. **输入 / 输出 / 验证**  
   展示一个用户或 agent 如何判断 skill 成功完成。

9. **真实案例**  
   只使用一个案例，不让案例抢走 skill 本身。

10. **可产品化判断**  
    判断它更适合个人 workflow、团队模板、开源项目还是企业工具。

11. **下一步**  
    未封口问题、后续优化、计划中的子功能。

### 6.2 通用展示原则

- 子页不是 `SKILL.md` 的网页复刻，而是讲“为什么这样设计”。
- 不展示长篇执行说明，不让页面像文档手册。
- 每页必须有真实挫折或迭代，否则会像静态工具宣传。
- 每页必须显示“产品化判断”，但第一版不展开商业化方案。
- 案例只服务于 skill 主线，不让案例变成主角。

## 7. 旗舰子页：reading-dialogue Skill

### 7.1 子页定位

路由：

`/ai/skill-desk/reading-dialogue/`

标题：

`reading-dialogue Skill：把深度阅读变成知识生产`

副标题：

一次真实长阅读任务让我意识到：深度追问如果没有边界，会从阅读沉淀滑进产品共创。于是我把“讨论、刹车、裁决、入库”做成了一套可复用 skill。

一句话定位：

这个页面讲的不是我读了什么书，而是我如何把深度阅读加工成一条可治理的知识生产 pipeline。

### 7.2 叙事主线

旗舰子页采用来源线程中确认的主线：

> 从一次失控的深度阅读里，长出一套有刹车、有门禁、有裁决的知识生产 skill。

重心顺序：

1. **思考**：为什么深度阅读不能直接等于总结，也不能直接等于制卡。
2. **实现**：`reading-dialogue` 如何把阅读加工成 pipeline。
3. **优化**：为什么后来加了档位、刹车、候选池、入库门禁和索引维护。
4. **收束**：它如何把“我为什么被这句话击中”变成未来任务能调用的知识资产。

`置身钉内` 只作为事故现场和优化来源，不作为页面主角。页面最多用 1-2 个模块说明它如何让 skill 暴露问题：多轮深聊从阅读讨论滑到企业 AI、责任流、AI 提醒小助手、V0、owner、风险池和指标。

### 7.3 内容矩阵

旗舰子页建议做成 10 个连续 section，可以沿用现有 `DeckLayout` 的沉浸式滚动，也可以做成普通长页中的 10 个方法模块。实现计划阶段再决定视觉形态。

| Section | 标题 | 核心内容 | 视觉建议 |
| --- | --- | --- | --- |
| S1 | 把深度阅读变成知识生产 | 常用技能入口。强调它不是读书总结器，而是阅读后的思考加工系统。 | pipeline 总图 |
| S2 | 原始问题：读完以后，想法会散掉 | 摘录保存了原文，但没有保存“我为什么被击中”“我想反驳什么”“我能迁移到哪里”。 | 摘录 vs 判断 对照 |
| S3 | 第一版直觉：多轮追问越深越好 | 围绕划线、批注、触动点追问，让读者从“这句重要”说到“它为什么对我重要”。 | 对话分叉 |
| S4 | 事故现场：《置身钉内》里的失控 | 多轮深聊从阅读讨论滑到企业 AI、责任流、AI 提醒小助手、V0、owner、风险池、指标。 | 讨论偏航图 |
| S5 | 关键反思：深度不是无限展开 | 真正的深度不是一直追问，而是知道此刻处在什么档位。 | 刹车系统 |
| S6 | 实现一：三档模式 | `划线交流档 -> 主题深挖档 -> 产品共创档`。进入产品方案时必须提醒用户切档。 | 三档状态机 |
| S7 | 实现二：追问刹车 | 单条划线最多两轮，主题深挖默认最多五轮。到点停靠确认：继续聊，还是先沉淀。 | 轮次计数器 |
| S8 | 实现三：候选池，不直接制卡 | 先生成阅读对话笔记和候选卡片池，区分用户原话、AI 推断、共创结论。 | 三栏归因表 |
| S9 | 实现四：入库门禁与卡片裁决 | 候选判断新增、追加、合并、暂缓、丢弃、留原文层；正式入库后维护索引。 | 闸门 / 裁决表 |
| S10 | 收束：读书变成长期判断力 | 价值不是替我读书，而是陪我把触动变成判断，把判断变成未来任务可调用的知识资产。 | 知识资产流入任务 |

### 7.4 关键文案

页面主表达：

`reading-dialogue` 的价值不是“替我总结”，而是“陪我把触动加工成判断”。

事故现场表达：

`置身钉内` 的作用不是展示我读了什么，而是暴露一个真实 workflow 问题：深度追问没有边界时，会从阅读沉淀滑到产品共创。

优化表达：

所以这个 skill 后来长出了三类控制机制：

- **档位**：划线交流、主题深挖、产品共创。
- **刹车**：单条划线最多两轮，主题深挖默认最多五轮。
- **门禁**：候选卡片先进池，再裁决新增、追加、合并、暂缓、丢弃或留原文层。

收束表达：

读书不是沉淀一堆摘录，而是让未来的 PRD、需求分析、原型审查、项目复盘能调用这些判断。

### 7.5 Pipeline

页面中应出现一条核心 pipeline：

```text
划线 / 批注
  -> 为什么触动
  -> 主题追问
  -> 最小记录
  -> 阅读对话笔记
  -> 候选卡片池
  -> 入库门禁
  -> 卡片裁决
  -> 正式知识域 / 原文层归档 / 能力层规则
```

这条 pipeline 需要在 S1 或 S6-S9 中重复出现，作为页面的视觉主轴。

### 7.6 输入、输出与验证

输入：

- 原文路径、文章、书籍、长文或阅读材料。
- 用户划线、批注、摘录、选中的片段。
- 用户阅读后的触动、疑问、反感、联想。
- 已结构化材料或已拆卡材料。

输出：

- 阅读对话笔记。
- 候选卡片池。
- 用户原话 / AI 推断 / 共创结论的区分。
- 入库门禁判断。
- 卡片裁决记录。
- 正式知识卡或暂缓候选。
- 材料索引、拆卡索引、正式知识域 README 的维护结果。

成功标准：

- 页面访客能在 1 分钟内理解它不是读书总结器。
- 访客能说出这个 skill 为什么需要档位、刹车和入库门禁。
- 作者自己能通过页面回忆 skill 当前边界和未封口优化点。
- 页面能自然连接到 `Personal Knowledge Harness`，说明这些知识资产最终进入三层知识库。

### 7.7 可产品化判断

`reading-dialogue` 第一版产品化判断：

- 当前形态：个人深度阅读 workflow。
- 团队复用潜力：高，适合团队读书会、行业研究、产品复盘后的观点沉淀。
- 开源潜力：中高，可抽象成 Markdown-first 的 reading dialogue pipeline，包括模板、CLI、Obsidian 目录结构和候选卡裁决表。
- 企业级潜力：中，适合企业知识管理、研发复盘、销售案例复盘、培训后沉淀，但需要补权限、多人协作、引用溯源和审核流。

第一版页面只展示判断，不承诺立即开源。

## 8. 与现有页面关系

`Skill Desk` 与现有 deck 的关系：

- `Personal Knowledge Harness`：总系统。说明 skill 最终如何落进三层知识库和多 agent 记忆系统。
- `从 Claude Code 到 AI 产品架构`：方法来源。说明作者为什么会用输入、工具、上下文、控制层、运行时看 AI 产品。
- `置身钉内：责任流阅读笔记`：一次产出。展示 reading-dialogue 加工后能产出什么产品判断。
- `Skill Desk`：技能方法库。展示这些产出背后的 skill 如何被设计、失控、修正和产品化。

建议互链：

- `/ai/skill-desk/reading-dialogue/` 结尾链接到 `/ai/zhi-shen-ding-nei/`，文案为 `查看一次真实阅读产出的产品判断`。
- `/ai/zhi-shen-ding-nei/` 结尾链接到 `/ai/skill-desk/reading-dialogue/`，文案为 `查看这套阅读方法如何被做成 skill`。
- `/ai/skill-desk/` 首页链接到 `Personal Knowledge Harness`，文案为 `这些 skill 最终进入个人知识 Harness`。

## 9. 视觉方向

首页视觉：

- 采用工作台感，而不是书架感。
- 信息密度高于普通作品卡片，但保持现有个人网站的轻量风格。
- 使用状态标签、成熟度标签、输出物标签和产品化标签，帮助快速扫描。

旗舰子页视觉：

- 不做传统书籍、笔记本、书架隐喻。
- 使用“加工流水线 + 刹车系统 + 闸门裁决”的视觉语言。
- 前半段：划线、批注、对话气泡、思考分叉。
- 中段：讨论流从阅读偏航到产品设计。
- 后半段：档位、刹车、闸门、裁决表、索引路由。
- 结尾：知识卡进入正式知识域，被未来任务重新调用。

色彩应避免和现有 deck 完全同质化。`Skill Desk` 可以更偏“工作台 / 控制台”，不使用大面积单一紫蓝渐变。

## 10. 非目标

第一版不做：

- 不把所有 skill 子页一次性写完。
- 不做完整开源项目页面。
- 不做可编辑后台或动态 CMS。
- 不把 `reading-dialogue/SKILL.md` 全文搬到网页。
- 不把《置身钉内》作为旗舰子页主角。
- 不重新解释整个三层知识库架构。
- 不在页面里展示内部路径过多，除非用于说明机制。

## 13. 风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 首页像工具目录 | 访客看不出产品设计能力 | 每张卡必须展示真实问题、成熟度、输出物和产品化判断 |
| 旗舰子页像 `SKILL.md` 文档 | 页面枯燥，缺少作品集表达 | 用“真实失控 -> 反思 -> 优化”的叙事结构 |
| 《置身钉内》抢走主角 | 变成阅读总结附录 | 只保留事故现场和优化来源，不展开责任流 deck 内容 |
| 产品化模块太重 | 第一版范围失控 | 只做判断，不做开源 README、仓库结构或商业方案 |
| 多个页面视觉割裂 | AI 栏目不统一 | 延续现有 Astro 静态页和 deck 语言，但给 Skill Desk 独立工作台气质 |
