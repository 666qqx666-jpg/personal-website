# 多业务线企业权限体系 Deck 设计 Spec

> 日期：2026-07-17
>
> 状态：设计已确认，等待用户复核书面 Spec
>
> 页面定位：个人网站「项目经历」第二个公开案例；对外合并展示两套已上线权限体系

## 1. 背景

个人网站的第一个项目经历 Deck 已用销售线索 SLM 证明产品演进、复杂机制编排和责任链重构。第二个 Deck 需要提供不同的能力证据：面对共用数据库和业务模块的两条业务线，不能因为共享底座就强行复用同一种成员与权限模型，而应根据真实治理对象进行差异建模。

项目内部包含两套相对独立、均已上线的权限体系：

1. 综合商场成员与权限体系，围绕商场范围和页面操作治理。
2. 连锁品牌组织与多维权限体系，围绕组织分支、门店网络、行政区、平台和页面操作治理。

网站对外将两者合并为一个案例、两个子项目。综合商场用于建立第一套清晰基线，连锁品牌承担主要深度，最终共同证明一个产品判断：共享业务底座，不代表必须共享同一种权限模型。

## 2. 已确认的核心决策

### 2.1 第一记忆点

访客看完后应能复述：

> 共享同一套业务底座，不代表必须共享同一种权限模型。

业务鉴权身份分离、审计操作身份统一，以及连锁品牌的五维权限设计，都是证明这项判断的机制，不与主线争抢。

### 2.2 案例组织方式

- 内部事实盘点仍将综合商场权限与连锁品牌权限视为两个项目。
- 网站展示为一个公开案例、两个子项目。
- 内容权重约为综合商场 30%、连锁品牌 70%。
- 综合商场负责建立简单、可解释的权限基线；连锁品牌负责展示差异建模深度。

### 2.3 叙事路线

采用「对照 → 分叉 → 收束」路线：

1. 先展示综合商场与连锁品牌两种不同的业务世界。
2. 解释两者为何继续共享数据库、业务模块和操作日志。
3. 以综合商场的二维权限模型建立基线。
4. 说明该模型为何无法直接覆盖连锁品牌。
5. 展开自定义组织树和五类独立权限对象。
6. 在高潮屏证明业务模块应按自己的数据对象读取权限，而不是把五类权限全局求交。
7. 用向下授权、同级隔离、总部治理和真实采用规模完成闭环。

### 2.4 产品判断高潮

全篇唯一高潮是：

> 五类权限不是必须同时满足的全局公式；每个业务模块应按自己的数据对象读取相应权限维度。

卡券读取门店和平台范围；销售线索读取行政区和平台范围；组织架构权限另外控制上下级成员及其创建的数据；页面与操作权限控制入口和具体动作。

### 2.5 结果证明

结尾使用「真实采用规模 + 治理结果」：

- 约 25 个综合商场。
- 约 30 个连锁品牌。
- 约 3,000 名已创建成员。
- 支持单客户 3 万多家门店的权限管理场景。
- 区域自治、同级分支隔离和总部跨区域治理已真实进入业务运行。

以上数据在页面实现前必须按项目总纲再次完成公开口径复核；页面继续使用“约”和区间表达，不包装成精确审计数字。

## 3. 目标与非目标

### 3.1 目标

- 让陌生访客先理解两条业务线管辖对象不同，再理解两套模型为何必须分叉。
- 证明本人能够识别权限对象、业务数据粒度、组织治理和共享底座之间的边界。
- 用五类权限的独立解释和模块按需取用关系，避免把案例写成后台配置项清单。
- 用已上线规模和真实数据隔离行为证明方案进入实际运行。
- 保持与销售线索 Deck 的明显体验差异，使不同案例分别证明不同产品能力。

### 3.2 非目标

- 不写成 RBAC、数据库或接口实现教程。
- 不完整公开生产权限字典、数据库字段、接口或可利用的鉴权规则。
- 不把两套业务权限强行合并为一个万能权限引擎。
- 不把综合商场和连锁品牌拆成两个公开页面。
- 不使用生产后台截图、真实组织名称或成员数据。
- 不使用 GSAP、ScrollTrigger、吸顶长编排或横向滚动轨道。
- 不修改销售线索 Deck、项目首页以外的内容页或 `DeckLayout` 全局滚动行为。

## 4. 受众与成功标准

### 4.1 主要受众

- 招聘方和业务负责人：判断候选人能否处理复杂 B 端组织与权限问题。
- 产品负责人：判断问题抽象、权限对象拆分和产品取舍能力。
- 研发与架构协作者：判断方案是否尊重数据对象、鉴权边界和复用成本。

### 4.2 内容成功标准

- 访客能说出为什么共用数据库不能直接共用成员模型。
- 访客能区分综合商场的二维模型与连锁品牌的五类权限对象。
- 访客能理解“模块按数据对象取用权限”与“所有权限全局求交”的区别。
- 访客不会误解为本人独立完成研发实现。
- 访客不会把规划、推测或未量化效率当成已验证结果。

### 4.3 体验成功标准

- 11 个场景按纵向顺序完整可读。
- 第二个 Deck 的体验稳定、清晰，区别于第一个 Deck 的连续演进和空间化滚动。
- 桌面端、移动端、减少动态和脚本失败状态均不丢失语义。
- 深浅色主题下，颜色、标签、边框和文字共同表达信息，颜色不是唯一编码。

## 6. 事实、状态与公开口径

### 6.1 本人角色

- 两个子项目均由本人完成问题识别、方案规划、业务建模、PRD、原型、技术协作和上线推动。
- 连锁品牌的自定义组织树、权限维度拆分、向下授权、前后端鉴权，以及业务身份与审计身份的关联方案，均由本人规划设计并推动上线。
- 对外表述为两套权限项目的完整产品负责人，不把研发和数据实现写成个人独立完成。

### 6.2 已上线事实

- 综合商场成员与权限体系已正式上线。
- 连锁品牌组织与多维权限体系已正式上线。
- 综合商场角色复用与商场组动态覆盖已进入实际使用。
- 连锁品牌区域自治、同级分支隔离、总部跨区域管理和前后端鉴权已进入实际使用。

### 6.3 结果边界

- 可以描述角色复用降低逐成员重复配置，但没有准确效率比例，不展示虚构的时间节省数字。
- 可以描述商场组自动覆盖未来新增商场，但不将其包装成组织效率实验。
- 可以描述同级隔离和总部治理真实成立，但不公开内部越权测试细节和完整鉴权策略。
- 采用规模证明系统真实运行，不将规模数字等同于单一功能带来的业务增长。

### 6.4 脱敏规则

- 客户、品牌、商场、组织节点、成员、门店和平台账号全部匿名。
- 图形中的组织树、商场、门店和平台使用虚构名称或抽象标签。
- 不公开生产权限字典、接口地址、数据库字段、内部角色清单或可被利用的鉴权细节。
- 所有业务图形使用 CSS/SVG 重绘，不加载生产后台截图。

## 7. 逐屏内容设计

### S1 封面：同一套系统，为什么需要两种权限模型？

- 标题直接建立反直觉问题。
- 副标题说明综合商场与连锁品牌共用数据库和多数业务模块，但真实治理对象不同。
- 不先使用 RBAC、成员表或接口鉴权等技术术语。

### S2 两种业务世界：商场管辖与门店网络

- 左侧综合商场：围绕具体商场、商场组、区域和集团岗位治理。
- 右侧连锁品牌：围绕门店网络、自定义组织分支、区域、事业部或品类治理。
- 两种世界上方分叉，下方共享数据库、业务模块和操作日志底座。

### S3 共同约束：共用数据库，不等于共用成员模型

- 公司不希望维护两套数据库，多数业务模块继续复用原有业务对象。
- 连锁品牌成员承担真实业务鉴权，并关联一个综合商场成员身份。
- 综合商场成员身份继续作为全系统统一操作身份写入操作日志。
- 本屏结论是“业务权限身份分离、审计操作身份统一”。

### S4 综合商场模型：管辖范围 × 页面操作

- 使用二维矩阵表达：`成员有效权限 = 商场管辖范围 × 页面操作范围`。
- 商场权限选择具体商场或商场组。
- 页面权限二选一使用角色或自定义权限。
- 角色只封装页面操作，不包含商场范围。

### S5 从逐人配置到角色复用与动态商场组

- 具体商场形成固定的绝对范围。
- 商场组形成随组织变化生效的相对范围，覆盖当前和未来新增商场。
- 常规岗位复用客户自建角色，特殊岗位继续使用自定义页面权限。
- 本屏控制篇幅，不展开完整角色清单。

### S6 模型分叉：品牌管的不是商场

- 连锁品牌核心管辖对象是企业组织和门店网络，不是商场。
- 客户既可能按区域组织，也可能按事业部或品类组织。
- 固定“总部—区域—门店”模板和综合商场权限模型均无法覆盖真实组织。
- 本屏承担从 30% 商场基线切换到 70% 品牌主线的转折。

### S7 五类权限对象：每一维解决一种治理问题

本屏使用一个完整篇幅分别解释五类权限：

1. **组织架构权限**：使用树状图，表达上下级成员治理及下级创建数据的管理关系。
2. **门店权限**：使用“全部门店 / 区域内门店 / 指定门店”三种范围图，解释以门店为数据对象的业务。
3. **行政区权限**：使用环形范围图，表达全部行政区、指定行政区和无权限；强调它直接作用于只携带行政区的数据。
4. **平台权限**：使用多平台分类图，表达成员可管辖的平台集合可被多个业务模块共同读取。
5. **页面及操作权限**：使用菜单树和操作按钮，表达页面入口与按钮级动作控制。

组织节点只确定上下级关系，不自动向节点内成员继承门店、行政区、平台或页面权限。

### S8 产品判断高潮：业务模块各取所需

- 五类权限不做全局求交。
- 卡券的业务范围读取门店和平台权限，行政区保持不参与。
- 销售线索的业务范围读取行政区和平台权限，不通过门店地址间接反推行政区。
- 组织架构权限另外决定成员的上下级治理关系，以及能否管理下级成员创建的卡券、线索等业务数据。
- 页面与操作权限作为入口和动作边界，继续作用于卡券、销售线索和成员治理。
- 门店、行政区和平台按业务数据对象选择；组织治理和页面操作以横向边界方式呈现，不与业务范围混成一个全局交集。
- 本屏结论是“权限越多不等于模型越强，关键是依赖关系与数据对象匹配”。

### S9 治理闭环：向下授权、同级隔离与审计统一

- 创建下级成员时默认带入操作者权限，只允许继续缩小范围。
- 操作者不能授予自己没有的门店、行政区、平台或页面权限。
- 同级分支互相隔离，总部保留跨区域治理能力。
- 前端控制入口和按钮，后端接口再次校验。
- 所有操作继续通过统一审计身份写入现有业务日志。

### S10 真实采用规模与治理结果

- 约 25 个综合商场。
- 约 30 个连锁品牌。
- 约 3,000 名已创建成员。
- 单客户 3 万多家门店场景。
- 区域负责人可以治理本区域成员及其业务数据，同级区域不能互相访问或修改，总部可以跨区域管理。
- 数字使用静态、克制的结果板，不使用庆祝式滚动数字。

### S11 结论：复用底座，而不是复用错误模型

- 收束到共享数据库、业务模块和操作日志继续复用。
- 强调两条业务线按自己的治理对象分别生长。
- 结尾不扩展到通用权限中台，也不暗示已经建设万能权限引擎。
- 提供返回项目经历页的清晰入口。

## 8. 视觉系统

### 8.1 整体风格

- 延续 `DeckLayout` 的极光背景、网格、玻璃面板和蓝紫青渐变。
- 跟随站点深浅色主题，不创建固定暗色孤岛。
- 不使用后台截图式界面，采用抽象但结构真实的 CSS/SVG 图形。

### 8.2 语义颜色

- 综合商场：青绿色，表达稳定、规整和范围矩阵。
- 连锁品牌：紫色，表达组织生长、分支和多维关系。
- 共享底座：蓝色，表达数据库、业务模块和审计复用。
- 关键判断：亮青色，强调 S8 模块读取关系。
- 越权阻断：玫红色，只用于边界与禁止路径。

颜色必须同时配合标签、边框形态和文字说明，不能成为唯一信息来源。

### 8.3 关键图形

1. S2–S3：两个业务世界位于同一共享底座上方。
2. S4–S5：商场范围与页面操作组成二维权限矩阵。
3. S7：树状组织、三档门店范围、行政区环形范围、多平台分类、菜单树与操作按钮。
4. S8：业务模块与五类权限之间的按需取用关系。
5. S9：组织树向下授权与同级分支阻断。
6. S10：静态规模与治理结果板。

## 9. 动效与响应式设计

### 9.1 与第一个 Deck 的体验差异

销售线索 Deck 使用 GSAP 连续滚动、两段 pin/scrub 和横向轨道，服务“探索、放弃、重组、反转”的产品演进故事。

权限体系 Deck 保留 `DeckLayout` 的整屏纵向阅读，每屏只解释一个权限概念，服务“两套模型对照与逐层拆解”。差异是内容适配，不是质量或完成度差异。

### 9.2 运行方式

- 不使用 GSAP 或 ScrollTrigger。
- 使用页面独立的 `IntersectionObserver + CSS` 动效控制器。
- 场景默认完整可见；脚本成功初始化后才添加动效准备状态。
- 单个场景只进行一次短时入场或状态点亮，不使用长时间 pin、scrub 或横向轨道。
- 不修改 `DeckLayout` 的全局 scroll-snap 行为。

### 9.3 重点动效

- S2–S3：共享底座保持稳定，两种治理世界分别展开。
- S4–S5：商场范围和页面操作先后点亮，商场组从固定集合扩展为动态范围。
- S7：五类权限图形按短时错峰依次进入，不锁定滚动。
- S8：卡券、销售线索和成员治理分别点亮需要的权限维度，无关维度保持弱化。
- S9：授权沿组织树向下流动，同级越权路径在边界处停止。
- S10：数据静态出现，不做庆祝式数字滚动。

### 9.4 移动端与减少动态

- 桌面端保持整屏导航和左右分栏。
- 移动端场景改为 `auto-height` 纵向排列，五维卡片和模块映射依次堆叠。
- `prefers-reduced-motion: reduce` 下关闭位移、错峰和状态过渡，直接展示最终状态。
- 动效不能成为理解内容的前提。

## 10. 技术结构

### 10.1 页面职责

`src/pages/projects/enterprise-permissions.astro` 负责：

- 11 屏语义化 HTML。
- 页面局部 CSS 和响应式规则。
- 脱敏后的 CSS/SVG 权限图形。
- 章节导航与返回项目页入口。

### 10.2 动效控制器职责

`src/scripts/enterprise-permissions-motion.ts` 负责：

- 建立和释放 `IntersectionObserver`。
- 根据场景进入状态添加稳定的 `data-motion-state` 或类名。
- 识别桌面端、移动端和减少动态偏好。
- 页面切换或卸载时停止观察并清理监听。
- 初始化异常时切回静态可读状态。

动画脚本通过稳定的 `data-scene`、`data-motion` 和 `data-motion-state` 属性选择元素，不将视觉类名同时作为脚本接口。

### 10.3 范围隔离

## 11. 降级与异常处理

| 情况 | 行为 |
| --- | --- |
| JavaScript 或动效脚本加载失败 | 所有正文和图形保持最终静态状态，页面完整可读 |
| 用户开启减少动态效果 | 关闭位移、错峰、点亮动画，直接展示最终状态 |
| 移动端或窄屏 | 取消固定整屏高度，所有图形纵向重排，无横向溢出 |
| 深浅色切换 | 仅更新 CSS 变量，不重新初始化 observer |
| 长组织名称或标签 | 允许换行和容器增高，不通过缩小字号隐藏溢出 |
| IntersectionObserver 不可用 | 不添加动效准备状态，保留静态内容 |
| 页面切换或卸载 | `disconnect()` observer，并清理媒体查询和事件监听 |

## 14. 风险与控制

### 风险一：案例退化为权限配置说明书

控制方式：S1–S6 先建立两种业务世界和模型分叉；S7 才解释五维；S8 用模块取用关系完成产品判断。

### 风险二：S7 五维信息密度过高

控制方式：每个维度使用独立图形和一句职责说明；桌面端并排，移动端纵向堆叠；不在同屏补充完整配置规则。

### 风险三：S8 关系图再次变得简陋

控制方式：S7 已完成概念建立，S8 只承担模块选取和横向治理边界；使用清晰连线、点亮状态和说明句，不再用孤立标签代替关系。

### 风险四：规模数字被误读为效率结果

控制方式：S10 将“采用规模”和“治理行为”分栏展示，不声称节省百分比或业务增长因果。

### 风险五：轻量动效被误解为完成度下降

控制方式：用更精细的图形层级和信息拆解建立质量；在项目经历整体复审时明确两个 Deck 的内容与动效适配逻辑。

### 风险六：动效初始化导致内容闪烁或隐藏

控制方式：HTML 默认最终可见，只有脚本成功后才添加准备态；任何初始化异常立即回到静态状态。
