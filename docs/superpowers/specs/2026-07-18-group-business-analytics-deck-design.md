> 日期：2026-07-18

> 状态：设计已确认，等待用户复核书面 Spec

> 页面定位：个人网站「项目经历」第三个公开案例；展示从集团经营考核需求、指标与统计建模，到数仓方案演进的数据产品规划能力

## 1. 背景

个人网站前两个项目经历案例分别证明了复杂经营机制演进与多业务线权限建模。第三个 Deck 需要补充另一类能力证据：面对集团管理层跨商场考核需求，如何把零散报表诉求转化为统一指标口径、集团—商场两级分析结构、可查询的数据对象，以及能够持续演进的数据方案。

这个案例不能写成四张报表的功能合集，也不能只用“至少 20 次下载变成一次查询”证明效率。访客还应看到本人能够完成一条完整的数据产品规划链路：

```text
集团管理问题
  -> 分析目标与指标口径
  -> 业务事实与统计粒度
  -> 第一代组合统计模型
  -> 使用中暴露的维护问题
  -> 基于明细 / 埋点对象的数仓模型
  -> 真实采用、技术债与持续规划
```

## 3. 定位与第一记忆点

### 3.1 对外名称

**集团经营数据分析体系**

### 3.2 一句话定位

把集团原本需要逐个商场查看、反复下载和人工汇总的会员与经营数据，转化为统一指标口径、集团—商场两级分析结构，并在真实使用中推动统计方案从组合结果表演进为基于业务明细与埋点对象的数仓模型。

### 3.3 第一记忆点

访客看完后应能复述：

> 数据产品不是先画一张报表，而是先明确谁用数据做什么判断，再定义指标口径、分析层级、事实对象和聚合粒度。

### 3.4 第二记忆点

> 第一代统计方案帮助需求快速落地；当维度和业务来源持续增长时，产品需要识别模型边界，并推动数据从预计算的组合结果回到可追溯的原子明细和行为对象。

## 4. 目标与非目标

### 4.1 目标

- 解释需求如何从集团管理层的真实工作方式中诞生，而不是从一个抽象 BI 平台规划开始。
- 证明本人能够把集团与商场的不同考核视角转化为指标口径、分析层级和数据对象。
- 用会员新增趋势与销售趋势解释第一代统计模型为何形成，再用小程序访问趋势说明数仓与埋点对象建模的演进。
- 让 S7–S8 的动画本身解释对象、粒度、查询维度和指标之间的关系。
- 如实呈现未迁移报表、未排期方案和未上线能力，证明持续规划能力而非包装完成度。
- 用约 25 个商场、集团季度考核和至少 20 次下载缩减为一次查询证明体系进入真实运行。

### 4.2 非目标

- 不做四类报表的逐功能展示或后台页面复刻。
- 不建设通用 BI、指标平台或新的数据仓库能力。
- 不把 MCUid 可关联多类数据写成已经上线统一漏斗。
- 不把会员新增趋势和销售趋势写成已经完成数仓迁移。
- 不公开生产表结构、接口、调度任务、客户名称或真实经营数据。
- 不把数据开发、数仓建设和工程实现写成本人独立完成。
- 不修改 `DeckLayout` 的全局主题和滚动行为。
- 不修改销售线索、企业权限或其他 Deck 的内容与动效实现。

## 5. 受众与成功标准

### 5.1 主要受众

- 招聘方和业务负责人：判断候选人是否能将集团经营问题转化为数据产品。
- 数据产品或产品负责人：判断指标定义、分析结构、事实对象和方案演进能力。
- 研发与数据协作者：判断方案是否尊重数据粒度、历史补算、扩展性和技术债边界。

### 5.2 内容成功标准

- 访客理解需求源于集团管理层需要统一查看和考核，而不是需要更多单商场报表。
- 访客能解释集团新增与商场新增为何使用不同口径。
- 访客能理解会员新增趋势与销售趋势如何形成第一代统计模型。
- 访客能区分日活记录、启动日志、页面访问日志和新增会员明细，并能把它们分别映射到 UV、VV、PV 和 NM。
- 访客知道数仓方案是后续演进，不会误以为四类报表从一开始就使用同一套数仓模型。
- 访客能说出仍未完成的迁移与未排期项。
- 访客不会误解为本人独立完成数据工程实现。

### 5.3 体验成功标准

- 11 个场景和 4 个章节按顺序完整可读。
- S7–S8 是全篇唯一局部吸顶段落，动效直接解释模型演进。
- 桌面端只创建一个 pin；移动端、减少动态、无 JavaScript 和初始化失败时均无 pin。
- 深浅色、390px 移动端和键盘导航下，文本、对象关系和状态标签都清晰可读。
- 动效、颜色和连线不是理解信息的唯一方式。

## 7. 事实、状态与公开口径

### 7.1 需求与结果证据

- 核心用户是集团管理层，需要用会员拉新与激活、积分运营、订单和销售数据考核下属商场。
- 报表上线前，典型的 10 个商场半年分析至少需要下载 20 次，并由集团人工合并。
- 集团—商场两级分析上线后，典型场景缩减为一次跨商场查询与下载，并可继续下钻单商场。
- 四类已上线报表作为综合商场底层标准能力，向约 25 个综合商场开放。
- 集团管理层已将会员数据纳入季度考核；订单数据用于完整销售额考核；积分数据用于 SP 活动与运营评价。

以上证据证明能力进入实际运行，不包装为严格效率实验，也不虚构节省比例或业务增长因果。

### 7.2 已上线能力

- 会员新增趋势 2.0。
- 小程序访问趋势。
- 积分产生 / 消耗趋势。
- 销售趋势。

四类报表分别呈现，没有已上线的统一跨模块转化漏斗。

### 7.3 方案完成但未排期

- 有价券分析：客户沟通、指标口径、数据方案、PRD 和原型已完成，但没有开发计划。

### 7.4 未排期技术债

- 会员新增趋势 2.0 的数仓改造。
- 销售趋势的数仓改造。
- 拍照积分订单虽已进入统一 SO 对象，但尚未纳入销售趋势。

### 7.5 本人角色与贡献边界

- 五类核心报表均由本人主导客户沟通、问题抽象、指标与分析维度建模、集团—商场两级结构设计、PRD、原型、研发和数据团队协作及上线推动。
- 本人会先研究源数据采集、分析对象、历史补算和数仓同步等技术方案，再交由研发与数据团队完成开发。
- 对外表述为完整负责数据产品方案和交付推动，不表述为独立完成数据开发、埋点开发或数仓工程。

## 8. 数据产品建模合同

### 8.1 会员新增趋势 2.0

业务口径先于数据结构：

- 集团新增会员：集团只统计真正首次开卡的会员。
- 商场新增开卡会员：会员首次在该商场开卡。
- 商场新增关联会员：已在集团其他商场开卡的会员，首次在该商场产生有效会员交互。
- 商场新增会员：商场新增开卡会员与新增关联会员之和。

第一代模型把日期、商场、新增类型、分布类型、分布来源和来源明细组成预聚合记录，并保存新增数量。业务事件负责增量更新，每日按日期和商场重新统计作为兜底。

公开页面只展示抽象后的维度与数量关系，不复刻生产字段枚举、通知接口或完整补算逻辑。

### 8.2 销售趋势

销售趋势从统一订单对象推导经营指标：

- SO 与负数 RSO 共同参与销售额计算。
- MCUid 用于销售人数去重和新老客判断。
- 商场 / 门店和日粒度支撑销售额、销售笔数、销售人数、连带率与老客率。
- 周 / 月指标需要尊重分子、分母和日数据的聚合规则，不能简单平均每日百分比。

第一代方案再按日期、商场 / 门店、数据来源和支付渠道等分析维度预先统计日指标，报表按周 / 月继续聚合这些结果。公开页面只解释“订单事实 → 日指标统计记录 → 周 / 月查询”的建模思路，不复刻未核验的生产统计表字段。

项目总纲确认销售趋势沿用类似的统计结果对象方案，尚未完成数仓改造。

### 8.3 第一代模型的价值与边界

第一代模型不是错误方案。它通过预先计算查询结果支持早期需求快速落地，并用每日重算降低网络或事务更新失败带来的数据不同步风险。

当商场、来源、推广渠道和业务维度持续增加后，边界逐步显现：

- 维度交叉产生更多组合统计记录。
- 业务事件更新可能遗漏、错乱或需要回溯补算。
- 空值也作为维度值参与组合，增加维护复杂度。
- 每日按商场与日期重算带来资源消耗和数据风险。
- 统计错误与修数工作频繁，新增平台或分析维度需要扩写更新链路。

### 8.4 小程序访问趋势的四类对象

数仓演进从“预先保存展示结果”转向“先保存可追溯的行为与业务明细，再按查询维度聚合”。公开版固定展示四类对象：

| 业务对象 | 事实粒度与职责 | 可公开分析维度 | 对应指标 |
| --- | --- | --- | --- |
| 小程序用户日活记录 | 用户在某平台、商场和日期的日活事实 | 商场、平台、日期、会员状态 | UV / 访问人数 |
| 小程序启动日志 | 每次启动及其来源、渠道和会员状态 | 商场、平台、日期、打开来源、推广渠道、用户类型 | VV / 打开次数 |
| 小程序页面访问日志 | 页面访问事实，并关联日活与启动上下文 | 商场、平台、日期、页面路径、上一个页面、页面明细 | PV / 页面访问量 |
| 小程序新增会员明细 | 开卡或绑卡事实及其转化与推广上下文 | 商场、平台、日期、新增类型、转化场景、推广渠道 | NM / 新增小程序会员 |

页面不得展示真实 appid、openid、XCUid、内部枚举、生产主键或可追溯到具体客户的页面路径。

### 8.5 查询层与指标层

数仓查询镜头使用公开维度：商场、平台、日期、用户类型、打开来源、推广渠道和页面路径。动画必须明确：

- 选择 UV 时聚焦日活记录。
- 选择 VV 时聚焦启动日志。
- 选择 PV 时聚焦页面访问日志。
- 选择 NM 时聚焦新增会员明细。

“按需聚合”表示查询时根据选择的指标和维度组合明细，不表示所有业务对象已经合并成一个通用宽表，也不表示所有旧报表已经自动迁移。

## 9. 逐屏内容设计

### S1 需求诞生：当管理对象从一家商场变成一个集团

- 不做纯标题封面，直接进入需求来源。
- 集团管理层不需要逐个进入商场页面完成集团分析；他们需要集团汇总、商场横向比较和统一考核入口。
- 视觉从多个分散商场数据面板收束为一个集团视角。
- 页面可用小字号标明项目名称“集团经营数据分析体系”。

### S2 从效率抱怨追到产品问题

- 典型的 10 个商场半年分析至少需要 20 次下载，单次最多三个月，再人工合并。
- “下载次数多”只是症状；核心问题是集团指标缺失、跨商场口径不统一、集团结果不能继续定位到具体商场。
- 本屏建立集团整体、商场对比和单商场下钻三层需求。

### S3 第一个需求：先定义会员新增，再设计统计表

- 展示集团新增、商场新增开卡、商场新增关联和商场新增会员四项口径。
- 高潮句：`集团新增会员 ≠ 各商场新增会员之和`。
- 从业务口径过渡到日期、商场、新增类型、来源维度和新增数量组成的组合统计记录。
- 不在本屏展开完整来源枚举。

### S4 第二个需求：从订单事实推导销售指标

- 从 SO / RSO、MCUid、商场 / 门店和日粒度推导销售额、笔数、人数、连带率与老客率。
- 用简化的指标关系说明同一订单事实在不同空间粒度和去重方式下产生不同指标。
- 强调比例指标在周 / 月聚合时使用分子与分母重新计算，体现数据分析口径判断。
- 把订单事实按日期、商场 / 门店、数据来源和支付渠道等维度转成日指标统计记录，再支持周 / 月查询；只展示统计表思路，不展示未核验的生产字段。

### S5 第一代方案：把查询结果提前算好

- 业务事件触发组合统计记录更新。
- 每日按单日、单商场重新计算作为兜底。
- 报表查询预聚合结果，以较低查询成本支持早期上线。
- 明确这是当时阶段的合理取舍，不写成错误反例。

### S6 使用中暴露边界

- 维度增加后组合节点不断增殖。
- 事件遗漏、空值组合、历史回溯和每日重算使更新链路复杂。
- 实际运行中出现统计错误、资源浪费和持续数据修复。
- 本屏结论：继续扩写组合结果表，会把每个新维度变成新的更新责任。

### S7 模型实验台上半场：行为如何落成业务对象

- 与 S8 共用桌面端局部吸顶舞台。
- 第一阶段：S6 的组合统计记录保留在画面中，错误与重算标记出现。
- 第二阶段：以抽象顾客行为驱动四类对象生成：进入小程序、再次启动、浏览页面、开卡 / 绑卡。
- 四类对象为日活记录、启动日志、页面访问日志和新增会员明细。
- 页面访问对象显式关联日活与启动上下文，但不展示生产主键。

### S8 模型实验台下半场：对象如何被投影为指标

- 第三阶段：查询镜头点亮商场、平台、日期、用户类型、来源和页面路径等分析维度。
- 第四阶段：NM、UV、VV、PV 分别连回对应事实对象，并生成简化趋势或排行结果。
- 高潮句：`先保留真实行为，再按问题选择聚合方式。`
- 动画表达模型设计的变化，不伪装成系统已经自动完成旧数据迁移。

### S9 产品反思：页面只是数据产品的最后一层

- 用短批注收束五个连续问题：谁用、做什么决策、指标怎么定义、事实对象是什么、按什么粒度聚合。
- 指出第一代预聚合帮助快速上线，后续数仓方案帮助扩展；两者对应不同阶段，而非简单的新旧优劣。
- 四类已上线报表只作为验证范围出现，不平铺功能。

### S10 技术债完整一章

- 使用状态矩阵明确列出：
  - 小程序访问趋势、积分趋势：已使用后续数仓方案。
  - 会员新增趋势 2.0、销售趋势：已上线但仍保留旧方案，数仓改造未排期。
  - 有价券分析：方案完成但未排期。
  - 拍照积分订单纳入销售趋势：未排期。
- MCUid 和统一 SO 只表示具备继续关联分析的基础，不表示统一漏斗已经上线。
- 技术债与未排期项必须常驻文本，不依赖 hover 或动效显示。

### S11 真实运行与本人角色

- 约 25 个综合商场开放。
- 集团会员指标进入季度考核，销售和积分数据进入相应经营评价。
- 典型场景从至少 20 次下载缩减为一次跨商场查询与下载。
- 明确本人负责客户沟通、指标与分析维度建模、数据方案研究、PRD、原型和交付推动；研发与数据团队完成工程实现。
- 结论：从集团考核需求，生长为可以持续演进的数据产品。
- 提供返回项目经历页的清晰入口。

## 11. 动效与响应式设计

### 11.1 普通场景

- S1–S6、S9–S11 使用原生 `IntersectionObserver + CSS` 一次性短入场。
- 不使用长时间 pin、scrub、横向轨道或庆祝式数字滚动。
- 场景默认完整可见；脚本成功后才进入动效准备态。

### 11.2 S7–S8 局部模型实验台

桌面端在不减少动态、视口宽度至少 1024px 时使用 GSAP Core 与 ScrollTrigger：

- 只创建一个局部 pin。
- 滚动距离目标约 180vh，具体数值在实现计划中以单屏可读性测试校准。
- 固定四阶段：组合表暴露问题、行为对象生成、查询镜头选择维度、指标映射收束。
- 不创建调试 markers。
- 模型实验台结束后恢复普通文档流，不影响 S9–S11。

### 11.3 移动端与减少动态

- 移动端将四阶段按纵向顺序完整展开，不创建 pin spacer。
- `prefers-reduced-motion: reduce` 下关闭位移、错峰、连线流动和 pin，直接显示四阶段最终状态。
- 无 JavaScript 时全部场景、对象、映射和状态文字可见。
- 对象关系在窄屏下从左右连线改为自上而下阅读，保持语义顺序。

### 11.4 动效状态机

| 模式 | 进入条件 | 视觉结果 |
| --- | --- | --- |
| `static` | HTML 初始状态或 JavaScript 禁用 | 全部内容最终可见，无 pin |
| `desktop` | 桌面宽度、未减少动态、GSAP 初始化成功 | 普通短入场 + S7–S8 一个局部 pin |
| `mobile` | 视口小于 1024px | 四阶段纵向展示，无 pin |
| `reduce` | 用户开启减少动态 | 全部最终可见，无位移、错峰或 pin |
| `fallback` | GSAP、ScrollTrigger、观察器或初始化失败 | 清理局部状态与 pin spacer，回到静态最终状态 |

## 12. 技术结构

### 12.1 页面职责

`src/pages/projects/group-business-analytics.astro` 负责：

- 11 屏语义化 HTML。
- 4 个章节导航和返回项目页入口。
- 局部 CSS、响应式和深浅色变量映射。
- 脱敏后的 CSS / SVG 数据对象、统计模型和经营图形。
- 稳定的 `data-scene`、`data-model-stage`、`data-model-phase` 和 `data-motion` 接口。

页面 frontmatter 集中声明章节、已上线报表、状态矩阵和公开规模，避免相同状态散落为多份文案真值。

### 12.2 动效控制器职责

`src/scripts/group-business-analytics-motion.ts` 负责：

- 注册和释放 GSAP ScrollTrigger。
- 只为 S7–S8 创建一个局部吸顶时间线。
- 用 `matchMedia` 切换 desktop、mobile 和 reduce 模式。
- 用 `IntersectionObserver` 管理其他场景的一次性短入场和章节导航状态。
- 在页面切换、卸载或媒体条件变化时 `revert()`、`kill()`、`disconnect()` 并移除监听。
- 初始化异常时移除准备态、清理 pin spacer，并设置 `fallback`。

脚本通过稳定的 `data-*` 属性选择元素，不把视觉类名当作脚本接口。

### 12.3 范围隔离

- 不修改 `src/layouts/DeckLayout.astro`。
- 不修改 `src/pages/projects/sales-lead-slm.astro` 和其动效控制器。
- 不修改企业权限 Deck 的内容、样式、控制器或测试。
- `.superpowers/` 仅保存 brainstorming 草稿，不进入正式提交。
- 项目总控指南只读，不在本 Deck 实施中改写。

## 13. 降级与异常处理

| 情况 | 行为 |
| --- | --- |
| JavaScript 禁用 | 11 屏、四阶段模型和状态矩阵全部静态可读 |
| GSAP / ScrollTrigger 加载或注册失败 | 不进入准备态或立即清理，设置 `fallback`，无残留 pin spacer |
| IntersectionObserver 不可用 | 普通场景保持最终可见，章节导航仍可点击 |
| 用户开启减少动态 | 关闭 pin、位移、连线流动和错峰，直接展示最终状态 |
| 移动端或窄屏 | 四阶段纵向排列，取消固定整屏高度和横向关系图 |
| 深浅色切换 | 只更新 CSS 变量，不重建模型时间线 |
| 快速滚动或反向滚动 | 模型时间线可逆且不丢失阶段正文，退出局部章节后恢复普通文档流 |
| 页面切换或卸载 | 清理 observer、media query、ScrollTrigger、时间线和事件监听 |
| 长标签或中文换行 | 容器自适应增高，不通过极小字号隐藏溢出 |

## 16. 风险与控制

### 风险一：案例重新退化为报表功能清单

控制方式：S1–S2 从集团管理问题开始；S3–S8 以口径、事实对象和模型演进为主；四类报表只在 S9–S11 作为采用范围出现。

### 风险二：S3–S8 过于技术化

控制方式：每个模型概念先回答业务问题，再展示数据结构；字段只保留商场、平台、日期、用户类型、来源和页面路径等公开维度。

### 风险三：动画被误解为自动数据迁移

控制方式：S7–S8 常驻说明“动画表达建模决策”；S10 独立列出未迁移报表和未排期技术债。

### 风险四：四类业务对象与指标映射不清

控制方式：四阶段动画末尾固定展示对象 → 指标一对一关系，移动端重复同一阅读顺序，测试直接锁定映射。

### 风险五：局部吸顶破坏移动端或快速滚动

控制方式：只在桌面增强模式创建一个 pin；移动、减少动态和失败模式无 pin；测试 pin 数量、清理和反向滚动。

### 风险六：公开过多生产细节

控制方式：只展示抽象对象和维度，不展示 appid、openid、XCUid、完整枚举、接口、任务调度和真实数据；图形全部脱敏重绘。

### 风险七：结果证据被包装成严格因果

控制方式：采用规模、季度考核和下载次数变化只证明真实使用与工作方式改变，不声称单一功能直接带来经营增长。
